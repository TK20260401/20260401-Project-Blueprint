## 概要（和文・約300字）

教育現場で16年蓄積した暗黙知は教室に閉じてきた。これをソフトウェア化する壁は、ドメイン知ではなくフルスタック実装力にあった。本報告は、専業エンジニアでない教育者が、生成AIエージェント（Claude Code等）を実装パートナーとして、要件定義から本番運用までを一人で貫通した実践を報告する。約2.5か月で複数の教育系Webアプリを設計・実装し、その一部を本番稼働させた。中心的な貢献は、（1）「要件のTo-be明文化→ロジックの純関数分離→AI実装→回帰テスト→コミット追跡」という再現可能な開発の型、（2）その過程で得た「AIに任せられる範囲と人間の検証が必須な範囲」の切り分け、(3)教育者ならではの非機能要件（全漢字ルビ・特別支援・失敗を罰しない設計）をAI駆動でどう実装に落としたか、である。学習効果の検証は今後の課題とし、本報告は方法論と実践知の共有に主眼を置く。

## Abstract（英文・要推敲）

Sixteen years of a teacher's tacit knowledge has remained confined to the classroom; the barrier to turning it into software has been full-stack implementation skill rather than domain knowledge. This practice report describes how a non-professional-engineer educator, partnering with generative AI agents (e.g., Claude Code), single-handedly carried multiple educational web applications from requirements definition to production within about 2.5 months. The contributions are: (1) a reproducible development "pattern" — to-be requirements, pure-function separation, AI implementation, regression testing, and commit-level traceability; (2) an articulation of which tasks can be delegated to AI and which require human verification; and (3) how an educator's non-functional requirements (full-kanji ruby, special-needs support, non-punitive failure design) were realized through AI-driven development. Learning-outcome evaluation is left as future work; this report focuses on sharing methodology and practical knowledge.

## 3. 方法：AI駆動開発の「型」（本報告の中心）

### 3-1. 5ステップ・ループ（図1として提示）

```
[1] 要件をTo-beで明文化   docs/requirements.md（P0/P1/P2・✅/🔄/🆕）
      │  ← 教育的妥当性・差別化軸を北極星として固定
      ▼
[2] ロジックを純関数に分離  lib/game/engine.ts 等（副作用ゼロ・乱数は注入）
      │  ← 永続化/認証/Realtimeは後付けにできる構造で手戻り防止
      ▼
[3] AIに実装させる        Claude Code で生成 → 人間が受け入れ判断
      ▼
[4] 自動テストで固定      vitest 回帰テスト
      │  ← 「直近実装の数え方を固定する」回帰の網
      ▼
[5] コミットで全工程を追跡  1機能=1コミット、履歴が設計判断の証跡
      └────── レビュー（AI多観点＋人間最終）→ [1]へ
```

### 3-2. 人間とAIの役割分担（表1）

| 工程 | 人間（教育者） | AIエージェント |
|---|---|---|
| 課題設定・教育的妥当性 | ◎ | △ |
| 要件定義（To-be明文化） | ◎ 優先度・差別化軸の決定 | ○ 文章化・抜け漏れ指摘 |
| アーキテクチャ設計 | ○ 方針決定（純関数分離等） | ◎ 具体案・トレードオフ |
| 実装 | △ 受け入れ判断 | ◎ 大半を生成 |
| テスト | ○ 観点指定 | ◎ テストコード生成 |
| レビュー・本番検証 | ◎ 最終受け入れ | ○ 多観点レビュー補助 |

要点： **意思決定・ドメイン知・最終検証は人間、実装速度・網羅性はAI**。この分担が崩れ「AI生成物を無検証で受け入れる」と品質が破綻する（6章）。

### 3-3. 実コードによる裏づけ（図2・図3）

- 純関数分離： `tetsu-board/lib/game/engine.ts`（乱数を引数注入 `rollDice(rng = Math.random)`＝テスト容易性とサーバ移行容易性を同時獲得）。
- 回帰テスト： `tetsu-board/lib/game/engine.test.ts`（`walk`/`judgeAnswer`/`calcScore` 等をvitestで固定）。
- **純関数化があるからテストが書ける**という依存関係を強調。

## 4. 実装事例（横断）

各事例を「何をAIに任せ、何が難しかったか」で記述（詳細は `AI-DRIVEN-DEVELOPMENT.md` 第5章）。

- **IPAS-Master（v9・本番）**: CRUD・グラフ・認証はAIが得意。難所＝14分野の分類体系と「たとえ話付き用語解説」の教育的設計。
- **Logic-Riichi（v30・本番）**: 待ち牌判定は純関数化と好相性でAI最適化の恩恵大。難所＝段階的足場外し（選択式→虫食い→自由記述）の学習設計とアクセシビリティ。
- **おこづかいクエスト（Web/Mobile）**: 9テーブル・サービス層をAIが高速生成。難所＝RLS・PIN暗号化など**安全性が壊れる層の検証**、Web/Mobile二重更新運用。
- **tetsu-board（2027目標）**: 本報告の型を最も体現（純関数engine＋vitest）。難所は技術より**法務・事業判断**（競合の無償教育版、独自化、地図出典）。

## 6. 考察

- **AI生成物の検証コスト**: 速いが正しいとは限らない。RLS・鍵・暗号化など「間違うと安全性が壊れる」層は人間検証が必須。純関数化＋回帰テストはこの検証コストを下げる工学的手段。
- **ドメイン知の非置換性**: AIは実装を増幅するが「何を作るべきか（教育的妥当性）」は人間のドメイン知が決める＝AIはドメイン専門家を**置換せず増幅**。
- **属人性↔再現性**: 独自性が高いほど再現性は下がる。本実践は要件・設計・テスト・履歴で再現性を補うが、完全再現は原理的に困難（AI出力の揺れ・人の判断依存）。

## 7. 限界と今後

- **限界**: 個人・実証段階。ユーザー数・**学習効果の統計的検証は未実施**。本報告の主張は「完遂できた／方法論を抽出できた」まで。
- **今後（第2弾への布石）**: 特別支援/通級等での**学習効果の評価研究**（教育工学領域）。製品化（おこづかいクエスト・Logic-Riichiは速やかに、tetsu-boardは2027）。倫理（子ども向け安全設計・AI生成物の知財）。
