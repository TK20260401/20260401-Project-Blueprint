## 0. プロダクト方針（要件の前提）

### ポジショニング
> 「麻雀の"何待ち"を、アルゴリズムで解いて鍛えるアプリ。AIに頼る前に、待ち読みの論理を自分の手で。」

### 勝ち筋（バズ調査から）
- 麻雀人気（Mリーグ・雀魂・麻雀AI MAKA）× アルゴリズム学習の交差点という稀有なポジション。
- 着火点は「麻雀 × エンジニア」層。本命は技術記事「待ち牌をビット演算で殴る」。
- **バズを定着に変える鍵は「動的OGPシェア画像」と「公開API」**。この2つを次期版の最優先機能とする。
- 技術記事（瞬発）× デイリー問題/ショート（持続）× 麻雀クラスタ（拡大）の三層で回す。

## 2. 非機能要件（Non-Functional Requirements）

### 2-1. 性能・可用性（バズ流入への耐性）
- **NFR-L-P01**: バズによるスパイク流入に耐える（SSG/エッジキャッシュ、OGP画像生成のキャッシュ）。
- **NFR-L-P02**: 公開APIはレート制限・濫用防止（IP/トークン単位）。コスト暴走を防ぐ。
- **NFR-L-P03**: OGP画像生成は遅延なく（生成結果をキャッシュ、初回以外は即返す）。

### 2-2. セキュリティ
- **NFR-L-S01**: 全回答結果に RLS、user_id 紐付け。
- **NFR-L-S02**: 公開APIは読取専用・書込不可、認証情報を要求しない範囲に限定。

### 2-3. アクセシビリティ／UD
- **NFR-L-U01**: 全漢字ルビ、牌画像 alt、色＋形の区別、`prefers-reduced-motion`、キーボード操作。

### 2-4. 運用・拡散容易性
- **NFR-L-O01**: シェア画像・OGP のデザイン品質（タイムラインでの視認性が拡散率を左右する）。
- **NFR-L-O02**: APIドキュメント・利用例の整備（記事から導線を張れる状態）。

## 3. 技術要件（Technical Requirements）

### 3-1. システム構成
- フロント＝Next.js（App Router, 16.x）/ React 19 / TypeScript / Tailwind CSS 4。
- グラフ＝Recharts（レーダー・棒グラフ）。
- BaaS＝Supabase（PostgreSQL + RLS + Auth）。
- ホスティング＝Vercel。

### 3-2. 新規技術要素（次期版）
- **TR-L01 OGP画像生成**: Next.js の動的OG画像（`ImageResponse` / `opengraph-image`）で牌姿を描画。生成結果はキャッシュ。
- **TR-L02 公開API**: Route Handler で待ち判定エンドポイント（手牌→待ち牌）。レート制限（Edge Middleware 等）。OpenAPI/README でドキュメント化。
- **TR-L03 待ち判定エンジン**: 既存の待ち牌計算ロジックを純関数として API・クイズ・OGP で共用（単一の真実源）。
- **TR-L04 自動ポスト**: デイリー問題のX投稿（Cron + 投稿連携）。

### 3-3. 外部連携
- **TR-L-E01**: X（共有・自動ポスト）／ハッシュタグ運用。
- **TR-L-E02**: （将来）画像認識サービス or 自前モデルで雀魂スクショ→牌認識。
