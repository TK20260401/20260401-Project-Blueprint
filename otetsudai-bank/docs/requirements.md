# おこづかいクエスト 要件定義書（次期版 / To-be）

> 本書は **Web版（otetsudai-bank）の正本**。モバイル版（otetsudai-quest-mobile）は本書の「共通コア要件」を共有し、別途プラットフォーム固有要件を `otetsudai-quest-mobile/docs/requirements.md` に定義する。
> 位置づけ: 現状（v0.4〜）からの **To-be（改善反映の次期要件）**。競合分析（`docs/` 配下の調査）で特定した差別化軸を要件として明文化する。
> 最終更新: 2026-06-19

---

## 0. プロダクト方針（要件の前提）

### ポジショニング
> 「カードを持つ前の、0歳目の金融教育。家庭内で安全に、お手伝い→報酬→つかう/ためる/あげるを"クエスト"で身につける。漢字が読めない子・特別支援の子も置いていかない、日本でいちばんやさしいお小遣いアプリ。」

### リリース方針（時期）
- **速やかに公開** を最優先とする。本体機能は v0.4 時点でほぼ揃っているため、次期版は「**公開ブロッカーの解消 → ストア公開 → 公開後すぐの改善**」の順で最小構成から出す。
- 公開ブロッカー（P0）を満たした時点で一般公開し、フリーミアム課金・週次レポート・特別支援拡張（P1）は公開直後の高速イテレーションで投入する。差別化軸は「最初から全部入り」ではなく「公開して回しながら厚くする」。

### 競合分析から確定した3つの差別化軸（全要件が従う北極星）
1. **特別支援・発達障害ファースト** — 全漢字ルビ・UDフォント・読み上げ・「失敗を罰しない」設計を中核価値とする。海外勢（Greenlight/Acorns Early）・国内記録型（ファミポイ等）のいずれも狙えていない最大の堀。
2. **カード非発行＝規制フリー＝安全** — 家庭内バーチャル残高（換金不可・対外決済不可・ユーザー間送金なし）に徹し、資金移動業・前払式支払手段の登録規制を**構造的に回避**する。これは設計原則であり、機能拡張の制約条件。
3. **日本語ネイティブのクエスト世界観＋金融教育** — 2022年金融教育必修化と日本の現金主義の親心理に合わせ、段階的にお金の概念を育てる体験を提供する。

## 2. 非機能要件（Non-Functional Requirements）

### 2-1. 法務・コンプライアンス（最重要・差別化軸②の制約）
- **NFR-L01**: コイン/残高は **換金不可・対外加盟店決済不可・ユーザー間送金不可** を製品原則として固定。これにより資金移動業・前払式支払手段（自家型/第三者型）の登録義務を回避する。
- **NFR-L02**: 上記原則を逸脱する機能（現金/ギフト券への換金、外部送金等）を検討する場合は、**実装前に資金決済法に強い専門家の確認を必須**とする（プロセスに組込み）。
- **NFR-L03**: 子ども個人情報の取扱い（保護者同意、最小限収集）、プライバシーポリシー・利用規約の整備・更新。
- **NFR-L04**: アプリストア審査の子ども向けカテゴリ規約（Kids Category）順守、外部リンク・広告・課金導線の制約に準拠。

### 2-2. アクセシビリティ / ユニバーサルデザイン（差別化軸①）
- **NFR-U01**: 全漢字ルビ（rt inherit方式でコントラスト自動調整）。
- **NFR-U02**: BIZ UDPGothic優先、最小タッチターゲット44px以上。
- **NFR-U03**: 色＋アイコン併用（色覚多様性）、`prefers-reduced-motion` 対応、フォーカスインジケータ・スキップリンク。
- **NFR-U04**: 文字サイズ多段切替・読み上げ対応（特別支援向け拡張）。WCAG AA以上を目標。

### 2-3. セキュリティ
- **NFR-S01**: 全テーブルRLS有効化、`get_my_family_id()` で家族単位アクセス制御。
- **NFR-S02**: PIN は bcrypt ハッシュ保存（平文非保持）。
- **NFR-S03**: privateデータを操作するAPI Route は service_role_key を使用し、anon_key を使わない（サーバ側限定）。
- **NFR-S04**: AIチャットへの入力に対するプロンプトインジェクション・不適切入力のガード（子ども利用前提）。

### 2-4. 性能・可用性・運用保守
- **NFR-P01**: 主要画面の初回表示が体感速い（モバイル回線想定、SSG/SSR最適化）。
- **NFR-P02**: Supabase（東京リージョン）共用構成での同時利用に耐える。
- **NFR-O01**: DB操作は `lib/services/` に集約し、コンポーネントから分離（保守性）。
- **NFR-O02**: 操作ログ／監査の蓄積（不正・障害調査）。

### 2-5. 操作性（UX）
- **NFR-X01**: 親の承認・タスク登録は最小操作（ワンタップ／テンプレ）。離脱要因の除去。
- **NFR-X02**: 子は読めなくても操作できる（アイコン＋音＋ルビ）。
- **NFR-X03**: レスポンシブ3段階（モバイル/タブレット/PC）。

## 3. 技術要件（Technical Requirements）

### 3-1. システム構成
- フロント＝Next.js（App Router, 16.x）/ React 19 / TypeScript / Tailwind CSS 4 / shadcn/ui。
- BaaS＝Supabase（PostgreSQL + Auth + RLS + pgcrypto）。東京リージョン、複数アプリ共用プロジェクト（`otetsudai_*` テーブル群）。
- ホスティング＝Vercel（CI/CD）。
- AI＝Claude API（ANTHROPIC_API_KEY、サーバAPI Route経由）。

### 3-2. データモデル（既存9テーブル＋拡張）
- 既存: `otetsudai_families` / `otetsudai_users` / `otetsudai_tasks` / `otetsudai_task_logs` / `otetsudai_wallets` / `otetsudai_transactions` / `otetsudai_spend_requests` / `otetsudai_badges` / `otetsudai_saving_goals`。
- 次期追加候補: 課金プラン（subscription）、招待（family_invites）、特別支援設定（accessibility_prefs）、週次レポート集計（report_snapshots）。新規テーブルは作成時にRLS必須化（GRANT/REVOKE方針順守）。

### 3-3. 外部連携
- **TR-E01**: 課金 — アプリストア内課金（モバイル）／Webは要検討（Stripe等）。フリーミアムの壁実装。
- **TR-E02**: Claude API（AIチャット・タスク提案・レポート文面生成）。
- **TR-E03**: メール/プッシュ通知（承認待ち・週次レポート・達成通知）。

### 3-4. プラットフォーム整合
- Web版（本書）とモバイル版はSupabaseバックエンド・データモデルを**共用**。ビジネスロジック・規制原則・UD方針は両版で同一に保つ（dual update）。差分はUIとプラットフォーム固有機能のみ。

## 4. スコープ外 / 将来検討
- 実マネー・カード発行・送金・換金（規制回避原則のため当面スコープ外。やるなら別事業体・専門家関与）。
- 自治体・教育機関への正式納品（法人化・契約・知財保証の整備が前提）。
