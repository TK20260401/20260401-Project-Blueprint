## 1. プロダクト概要

**名称**: おこづかいクエスト（Otetsudai Quest）
**種類**: 家族向けマネー教育 PWA（将来的にネイティブアプリ展開予定）
**コンセプト**: お手伝い＝クエスト。子どもが冒険感覚で家事をこなし、「稼ぐ・貯める・増やす・使う」を体験するマネー教育アプリ。
**現状バージョン**: v0.9.2（ほぼ全機能実装済み、ベータテスト段階）

### 技術スタック
- Next.js 16 / React 19 / Tailwind CSS / shadcn/ui
- Supabase（PostgreSQL + Auth）
- Claude API（AIアシスタント）

### 3ロール体制
- **子ども**: PIN認証、クエスト完了報告、貯金目標、投資シミュレーション
- **親**: メール認証、クエスト管理、承認ワークフロー、支出承認
- **管理者**: お知らせ、株価マスター管理、メンテナンスモード

## 2. 対象ユーザー

| セグメント | 年齢 | UI 要件 |
|------------|------|---------|
| 未就学児 | 4〜6歳 | ひらがな中心、大型タッチターゲット、親の補助前提 |
| 小学校低学年 | 6〜9歳 | ルビ必須、直感的アイコン、即時フィードバック |
| 小学校高学年 | 9〜12歳 | 自走可能なUI、ゲーミフィケーション強化 |
| 特別支援学校中学部 | 〜15歳 | UD フォント、構造化レイアウト、ルビ全漢字、感覚過敏配慮 |
| 特別支援学級・通級 | 〜15歳 | 段階的利用、視覚的シンプルさ、操作エラーへの寛容性 |

**重要な設計制約**:
- **読みやすさは必須要件**（ルビ振り・色＋アイコンの二重コーディング・明確な視覚階層）
- **認知負荷を抑える**（一画面に情報を詰め込まない）
- **自己効力感の醸成**（「できた！」体験の最大化）

## 3. 指導を仰ぎたい領域

### A. UI デザイン（画面単位）

以下の画面について、デザイン観点でレビューをお願いしたい：

1. **ランディングページ** (`/`)
   - 3本柱（使う・貯める・増やす）の訴求力
   - 保護者の初回導線（サインアップ CTA の配置・文言）

2. **子どもダッシュボード** (`/child/[childId]`)
   - レベル表示・キャラクター・クエスト一覧・ウォレット（3分割）・バッジ・投資シミュレーションが1画面に集約
   - 情報量が多すぎないか？認知負荷バランスはどうか？
   - タブ構造（クエスト/履歴/投資）の最適解は？

3. **親ダッシュボード** (`/parent`)
   - 承認待ち・子ども残高・統計の並び順
   - 承認ダイアログのフロー（スタンプ・メッセージ・却下理由）

4. **管理者ダッシュボード** (`/admin`)
   - 業務ツールとしての情報密度（現状やや詰め込みすぎ？）

### B. UX（タスク完遂体験）

1. **子どもの初回体験**
   - PIN入力 → クエスト選択 → 完了報告 → 承認待ち → 報酬
   - 「承認待ち」の時間を子どもがどう受容するか（即時性の欠如への配慮）
   - 初回ログイン時のオンボーディング（現状ほぼなし）

2. **親の承認ワークフロー**
   - 通知手段（現状はダッシュボード表示のみ。プッシュ通知なし）
   - 複数の承認要素（クエスト／支出／投資／じぶんクエスト）が混在する際の優先順位
   - 「やりなおし」を伝える際の文言と感情配慮

3. **報酬分配の理解**
   - 3分割スライダーの意図（使う・貯める・増やす）が親子に正しく伝わるか
   - 教育的説明（UI に対するマイクロコピー）

### C. CX（長期的な顧客体験）

1. **習慣化の仕組み**
   - 毎日／毎週繰り返しクエストの UX
   - レベルアップ・バッジ獲得の報酬設計（内発的動機付けの維持）
   - 「使わなくなる理由」の予防

2. **家族コミュニケーション**
   - スタンプ・メッセージ機能が親子の会話を促進できているか
   - チャット機能（親子間）の位置づけ（現状 AI チャットのみで親子間は未実装）

3. **教育的 CX**
   - 「投資で値下がり」の受容体験（実損はないが心理的ダメージのケア）
   - 「じぶんクエスト却下」時の子どものメンタル保護
   - 「貯金目標未達」時のフォロー設計

## 4. 現時点で認識している課題

| 課題 | 現状 | 仮説 |
|------|------|------|
| ルビによる行高不揃い | globals.css で line-height: 1.75 統一（v0.9.2で対応済み） | 他に視覚的ガタつきがないか要レビュー |
| 一画面の情報密度 | 子ども画面に機能が集約 | タブ化で緩和中。さらなる分離が必要か？ |
| プッシュ通知なし | PWA の Web Push 未実装 | 親の承認遅延の主因。優先度高？ |
| アクセシビリティ検証不足 | WCAG 準拠の検証未実施 | コントラスト・フォーカスリング・スクリーンリーダー対応の第三者監査希望 |
| マイクロコピー統一感 | 「〜しよう！」「〜できます」が混在 | 全ページ トーン＆マナー定義必要 |
| ダークモード | CSS 変数は用意されているが未検証 | 子どもの使用時間帯（夜の学習時）に必要か？ |
| キャラクターの魅力 | 絵文字のみで独自キャラクター不在 | ブランディング・愛着形成の観点で課題 |

## 5. ご相談したい具体的な問い

**デザイナーへの質問（優先度順）**:

1. 対象ユーザーの幅が広い（4歳〜中3・特別支援）。**単一 UI で包含すべきか、モード切替（やさしい／ふつう）を導入すべきか？**
2. 子どもダッシュボードの情報階層を再設計するとしたら、**最優先に見せる情報は何か？**
3. 「承認待ち」の非同期体験を、子どもが受容しやすくする **UI／マイクロコピーのパターン**は？
4. ゲーミフィケーション（レベル・バッジ・キャラクター）を、**教育的に負の側面（競争・他者比較）なく設計する原則**は？
5. PWA としての体験（ホーム画面追加・オフライン対応・通知）を、**ファミリー向けアプリで最低限確保すべきライン**は？
6. ブランドアイデンティティ（ロゴ・カラー・タイポグラフィ・キャラクター）を**ゼロから構築する場合の工程**を教えてほしい。
7. 特別支援教育の観点から、**感覚過敏・認知特性に配慮した UI 設計のベストプラクティス**は？

## 8. 最終ゴール

「**教育的に正しく、使っていて楽しく、家族の会話が生まれる**」プロダクトへ磨き上げること。特に、

- 「おこづかい帳」ではなく「家族の冒険ログ」と感じられる体験
- 子どもが自走し、親が適切に関与できる CX 設計
- 特別支援教育の現場でも安心して採用できる UX 品質

これらを実現するためのデザイン視点のご指導をいただけますと幸いです。
