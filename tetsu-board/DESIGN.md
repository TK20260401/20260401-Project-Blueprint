## 1. プロジェクト概要

### 1.1 一行コンセプト

> **授業のスキマで遊べる、5〜15分のクイズ駆動・視覚優先のボードゲーム。
> 知識定着がそのまま勝敗になる。仲間のプレイを見て学べる。**

### 1.2 戦略的位置づけ

- **個人開発の小規模ゲーム** ではなく **教育プロダクト** として設計する
- 鉄道ボードゲーム先行作品の骨格を借りつつ、教育用途に最適化した独自路線
- 市場に類似品ほぼなし(特に第1弾ターゲット層には競合不在)
- 開発期間目安: **6ヶ月〜1年**(個人開発)
- 完成後の展開可能性: 教育委員会・特支学校・通信制高校への提供、教員向け教材販売

### 1.3 デュアル成立目標

| 軸 | 達成すべきこと |
|---|---|
| 教育的に成立 | 教材として現場で使える(教師ガイド・評価ルーブリック完備) |
| 娯楽として成立 | 個人で遊んでも面白い(余暇モードで20〜30年プレイできる手応え) |

## 2. ターゲットと製品戦略

### 2.1 第1弾ターゲット: C層

**知的障害・発達系を含む特支/夜間定時制の生徒**

選定理由:
1. 開発者(菊池)の現場経験が最も濃い領域
2. 当事者性(ASD/ADHD)が設計に直接反映される
3. 競合がほぼ存在しない(差別化最大)
4. C向けに作った合理的配慮UIは、A/B層にも転用可能(下方互換性)

### 2.2 拡張ロードマップ

```
第1弾: C層向け(本ドキュメントの対象範囲)
  ↓
第2弾: A層(小学生〜中学生)モード追加
  ↓
第3弾: B層(高校生)モード追加
  ↓
第4弾: 大人/リスキリング層、教員自身向け
```

各層への展開は **UIスキンとコンテンツパックの差し替え** で実現する(コアエンジンは共通)。

### 2.3 教材性のレベル

**「濃い」レベルで作る**:
- 教師用ガイド
- 授業設計案(45分単位、複数単元)
- 評価ルーブリック
- 振り返りシート(児童/生徒用)
- 学習目標→ゲーム要素マッピング表
- 教師向けプレイヤーズガイド

## 3. ゲームデザイン (Phase 1 確定範囲)

### 3.1 鉄道ボードゲームジャンルからの継承要素

骨格として以下を採用:

- ✅ サイコロで線路上を移動
- ✅ 駅で物件を取得
- ✅ ランダムに **目的地** が設定される(1着ボーナス、最下位ペナルティ)
- ✅ 何らかの **災難キャラ**(災難キャラクター(独自キャラ))が最下位に憑く
- ✅ **カードシステム**(急行/防御/移動系など)
- ✅ **N年プレイ後の総合スコアで勝敗**(期限制)
- ✅ 決算タイミング(年度末)
- ✅ 物件 **独占ボーナス**(同一駅の全物件取得で収益増)

### 3.2 独自要素(先行作品(鉄道ボードゲーム)から離れる4点)

#### ① 勝敗の通貨が「お金」から「正解」に変わる

- 主スコア = **クイズ正解数**
- 副スコア = コイン・物件
- 終了時スコア式: `正解数 × 重み + 資産 × 重み`

#### ② クイズ駆動のコアループ

- 駅に到着 → クイズ出題
- 正解 → コイン + 物件取得 + 演出
- 不正解 → ヒント表示 → 次へ(罰則は最小限、自己効力感重視)

#### ③ 「みんなで見る」前提の同期型設計

- プレイヤーが操作していない時間も「見て学ぶ」が成立する
- 他者の正解/誤答を観察することが学習機会になる

#### ④ 合理的配慮を最優先するUI(後述「5. 合理的配慮原則」)

### 3.3 勝利条件: 知識定着テスト型

**クイズ正解数を主スコアとした総合得点制**

- ゲーム終了時に「正解数 + 資産」を合算
- 配分の重み付けは要調整(Phase 3で決定)
- 勝敗だけでなく **「何を学んだか」が可視化** される設計

### 3.4 プレイモード

| モード | プレイ時間 | ターン数(目安) | 用途 |
|---|---|---|---|
| **アイスブレイク** | 5〜15分 | 12〜36ターン(1〜3年) | 授業冒頭、スキマ時間 |
| **余暇(エンドレス)** | 〜1時間 | 240〜360ターン(20〜30年) | 放課後、休み時間、通信制でじっくり |

両モードとも **セーブ・再開機能を必須** とする(余暇モードは特に途中離脱が前提)。

### 3.5 クイズと経済システム: a) クイズ報酬制で確定

```
プレイヤーが駅に到着
  ↓
クイズ出題(教科・難易度はランダム or 単元設定に従う)
  ↓
正解 → 🪙コイン獲得 + 🏠物件取得 + 演出
不正解 → ヒント表示 → 軽微な進行(再挑戦 or 次のプレイヤーへ)
  ↓
画面上部に「正解数 / コイン / 物件数」が常時表示
```

- **コインは絵で表現**(数字ではなく100円玉100枚の絵で「100円」を示す)
- **物件は建物のアイコン**(産業種別が一目でわかる)
- お金の概念は **具体物として体験的に理解** させる

### 3.6 コアループ(1ターンの流れ)

```
[自分の番が回ってくる]
  ↓
サイコロを振る(アニメーション演出)
  ↓
線路上を駒が動く(各マスを経由する動き)
  ↓
駅に到着 → クイズ出題
  ↓
正解判定 → 報酬演出 / 不正解時はヒント
  ↓
災難キャライベント・カード使用などのオプション
  ↓
ターン終了 → 次のプレイヤーへ(バトン演出)
  ↓
[N年経過 → 決算 → スコア更新]
```

## 4. マップと移動 (Phase 2 確定範囲)

### 4.1 駅プールの方針

**完全架空(カイロソフト風)+ 実在駅を括弧で副表示(教育補強)**

- 主表示 = 親しみやすい架空駅名(例: 「みかんばたけ駅」)
- 副表示 = 実在駅名・都道府県(例: 「(愛媛・松山駅)」)
- 児童・生徒は架空名で覚え、慣れてきたら実在駅と結びつける

#### 表示モード(ハイブリッド)

- **C層初期値**: 副表示OFF(架空名のみ)
- **A/B層初期値**: 副表示ON(常時併記)
- **タップ操作**: いつでも実在駅情報を表示可能
- **教師アカウント**: クラス全員に一括設定可

### 4.2 マスター駅プール構成

| カテゴリ | 駅数 | 産業構造との対応 |
|---|---|---|
| 田畑・里山系 | 15 | 第一次産業 |
| 海・港系 | 10 | 第一次産業 |
| 工業・ものづくり系 | 10 | 第二次産業 |
| 都市・繁華街系 | 10 | 第三次産業 |
| 観光・自然系 | 10 | 第三次産業 |
| イベント駅(5種類×複数) | 10〜15 | ゲーム機構 |
| **合計マスタープール** | **65〜70駅** | |

#### 駅カテゴリと産業構造の対応

カテゴリ分類は **社会科の産業構造学習(第一次/第二次/第三次)** と一致させる。
将来A/B層展開時に「産業構造を学ぶ単元」として活用できる構造。

#### イベント駅(5種類)

| 駅名 | アイコン | 効果 |
|---|---|---|
| クイズの駅 | ❓ | クイズイベント発生 |
| ボーナスの駅 | 💰 | コインが必ずもらえる |
| カードの駅 | 🃏 | カード1枚獲得 |
| ワープの駅 | 🌀 | マップ上を瞬間移動 |
| ピンチの駅 | ⚠️ | 災難イベント(災難キャラ(独自)憑依等) |

#### 駅名サンプル

60駅の駅名案は別途記載(本文書末尾の付録または `STATION_POOL.md` で管理予定)。
**実在駅マッピング作業は Phase 6 で実施**(本フェーズでは方針確定のみ)。

### 4.3 1ゲームの駅数

- **1ゲームあたり 15〜20駅**(両モード共通マップ)
- マスタープール(65〜70駅)から毎回ランダム選定
- 物件・クイズ問題も毎回シャッフル(ローグライク要素)

#### 両モードの駅数共通の理由

鉄道ボードゲーム先行作品も同じマップを3年/50年の両方で遊ぶ。マップは固定、ターン数で長短を制御するのが王道。

### 4.3.1 マップ自動生成アルゴリズム(E-25 確定、v2.0)

#### 基本: テンプレ駆動 + 制約付きランダム(児童モード標準)

- **マップ形状はテンプレ固定**: 円環+3分岐の構造、駅配置位置18個+イベント駅5種
- **駅選定**: マスタープール65〜70駅から18駅をランダム選定
- **カテゴリバランス保証**: 5カテゴリ(田畑/海/工業/都市/観光)各3〜4駅
- **イベント駅5種**: 円環上にランダム配置、隣接禁止制約
- **分岐パターン割当**: 4パターン(近道&危険 vs 遠回り&安全 等)を3分岐位置にランダム割当(パターン重複OK)
- **シード固定機能**: 同じシードで同じマップを再生成(教師の比較学習・授業の続きに有効)

#### 拡張: 単元連動(教師モードオプション)

- 教師が単元を選択(例: 第一次産業/サービス業/工業)すると、関連カテゴリの駅を優先選定
- 教科横断モード(Phase 5確定)と整合
- 単元別の駅プール優先割合は授業設計に応じて教師が調整可

#### ローグライクシード設計

- マップID = `YYYY-MM-DD_クラスID_セッションID`
- 児童モード: 毎回新規シード
- 教師モード: シード固定可、過去マップ再現可
- 同一シード = 駅選定+配置+物件+分岐パターンすべて再現

### 4.4 マップ形状: D) 円環+少数分岐

```

        │     │       │
```

## 5. 経済・スコア (Phase 3 確定範囲)

### 5.1 物件の種類と効果

#### 5.1.1 駅カテゴリと物件種は1対1対応

| 駅カテゴリ | 物件種 | 例 | 産業 |
|---|---|---|---|
| 田畑・里山系 | 🌾 畑物件 | りんご畑、いちご畑 | 第一次 |
| 海・港系 | 🐟 漁港物件 | かに漁、まぐろ漁 | 第一次 |
| 工業系 | 🏭 工場物件 | ねじ工場、ガラス工場 | 第二次 |
| 都市系 | 🏪 お店物件 | レストラン、洋服屋 | 第三次 |
| 観光系 | 🏨 ホテル物件 | 温泉宿、リゾートホテル | 第三次 |

各駅に **2〜4種類の物件** が用意されている。

#### 5.1.2 価格・収益のティア構成

整数のみ・全駅で同じスケール。

| ティア | 価格 | 年間収益(決算時) | 元取れ年数 |
|---|---|---|---|
| 小 | 10コイン | 2コイン | 5年 |
| 中 | 30コイン | 5コイン | 6年 |
| 大 | 100コイン | 12コイン | 8〜9年 |
| 特大 | 300コイン | 30コイン | 10年 |

#### 5.1.3 表示方針: 整数+%併記オプション

**「漢字+ルビ」と同じくトグル切替方式**で運用:

```
[OFF: 合理的配慮モード]            [ON: 詳細モード]
┌────────────────┐              ┌────────────────┐
│ 🌾 りんご畑     │              │ 🌾 りんご畑     │
│                │              │                │
│ かう: 30コイン  │   ⇄          │ かう: 30コイン  │
│                │              │                │
│ もらえる:       │              │ もらえる:       │
│ 5コイン         │              │ 5コイン (15%)   │
└────────────────┘              └────────────────┘
```

- **C層初期値**: OFF(整数のみ)
- **A/B層初期値**: ON(%併記)
- アクセシビリティ設定で **ルビON/OFFと同じ場所** で切替

#### 5.1.4 独占ボーナス

```
[同じ駅の物件をぜんぶ買った時]
独占ボーナス: 収益が 2倍 になる!
```

- **2倍固定**(分数・複雑な倍率は禁止)
- 演出: 駅の上に「🏆 どくせん!」アイコンが点滅

#### 5.1.5 物件成長は廃止

- 従来作では時間経過で物件価値が上がるが、 **これは可視化困難** のため第1弾では廃止
- 物件価値・収益は **時間経過で変わらない固定値**
- 「物件を集める喜び」は **独占ボーナス** で担保

#### 5.1.6 物件売却機能(E-21 確定、v2.0)

- 売却タイミング: **物件のある駅に止まった時のみ** (自駅売却)
- 売却額: **購入価格の50%**
- 確認モーダル1段階(「ほんとうに売る? はい / いいえ」)
- 売却後、独占ボーナス対象から外れる
- 売却UIは駅メニューから「うる」ボタンで選択
- C層配慮として「タップ → 駅メニュー → うる → 確認」の3ステップで衝動売却を抑制

### 5.2 収益モデル

コインが増える経路は **3系統 全部採用**:

#### 5.2.1 クイズ正解報酬(主軸)
- 駅でクイズ正解時にコイン獲得
- 即時に演出+コイン加算

#### 5.2.2 通過駅コイン
- 駅を通過するたび小コイン(1〜2枚)
- 「ピッ」と音とともに獲得

#### 5.2.3 物件収益: 決算時+訪問賃料(ハイブリッド)

| 経路 | 発生タイミング | 額 |
|---|---|---|
| **決算時収益** | 年度末(12ターン目) | 物件価格の **1/5** |
| **訪問賃料** | 他プレイヤーが物件駅に止まる時 | 物件価格の **1/20** |

- 決算時 = 大きなボーナス、年に1回の盛り上がり
- 訪問時 = 小さな賃料、リアルタイムで「コインが入る!」演出
- 両方併存で、視覚的快感+観察学習機会を最大化

### 5.3 スコア重み付け: 完全分離3部門制

**スコアを統合せず、別々の指標として表示**

#### 5.3.1 3部門の表彰

- 🏆 **クイズ王**(正解数1位)
- 💰 **金持ち王**(資産1位)
- ⭐ **総合王**(正解数+資産の合計1位)

#### 5.3.2 完全分離の意義(C層特に有効)

- **多様な評価軸**で「自分の良いところ」が見つかる
- 計算が苦手な児童・生徒も「正解で1位」になれる
- 物件運営が得意な児童・生徒も「金持ちで1位」になれる
- 自己効力感を最大化(特支教育の核)
- A/B層展開時も同じ仕組みで対応可能

### 5.4 決算の具体仕様

#### 5.4.1 1年=12ターン

- カレンダー1ヶ月=1ターンの対応で月の概念学習との連動
- 家庭科の食材旬・季節学習にも活用可能

| モード | ターン数 | 年数 |
|---|---|---|
| アイスブレイク | 12〜36ターン | 1〜3年 |
| 余暇 | 240〜360ターン | 20〜30年 |

#### 5.4.2 決算前の年間まとめクイズ

- 1年の総まとめ問題を出題
- 全員正解でボーナス(教育的振り返りの瞬間)
- 演出時間: 1〜2分、教師スキップ可

#### 5.4.3 決算演出の流れ

```
1. 「3がつ おわり! けっさん!」演出(2秒)
    ↓
2. 年間まとめクイズ
    - 1年の総まとめ問題
    - 全員ボーナス可
    ↓
3. 各プレイヤーごとに今年の収益発表
    - 「あなたの ものけん: りんごばたけ × 3」
    - 「コイン 15まい!」
    - コインが画面に転がる演出
    ↓
4. 順位発表
    - 1位 ⭐⭐⭐ / 2位 ⭐⭐ / 3位 ⭐
    - AIキャラのコメント(「やったね!」「がんばったね!」)
    ↓
5. スコアボード更新
    - クイズ王・金持ち王・総合王の3部門それぞれ更新
    - 累計スコアが上書き
```

- 演出時間: **30秒〜1分**(教師スキップ可)
- 全プレイヤー名+今年の成績が一覧で見える
- **「ぜんぜん正解しなかった」プレイヤーも何かしら褒められる演出**(合理的配慮)

## 7. クイズシステム (Phase 5 確定範囲)

### 7.1 設計原則(新規追加): イニシアチブはユーザ

すべてのインタラクションで、 **児童・生徒(=ユーザ)が主導権を持つ**。教師は監督・必要時介入・枠の設定者。

この原則は Phase 5 で明文化されたが、Phase 4 以前のすべての決定にも貫通する。具体的な現れ方:

- 動的難易度: 自動調整 + ユーザがいつでも上下調整可
- 制限時間: ユーザが「ゆっくり/ふつう」を選択
- ヒント・たすけ: ユーザが「ほしい」と能動的に求める
- もう一度挑戦: ユーザが選んで再挑戦
- 学習接続: 自分の「ふくしゅうリスト」が見える

### 7.2 出題ロジック(教科×駅の関連付け)

3モード切替制で運用:

| モード | ロジック | 用途 |
|---|---|---|
| **完全対応モード** | 駅カテゴリ=教科の固定対応 | 単元学習 |
| **弱い相関モード** ★初期値 | 70%関連 + 30%ランダム | 通常プレイ |
| **教科横断モード** | 完全ランダム | 総合復習 |

教師がモード選択。初期値=弱い相関モード。

### 7.3 難易度: 動的調整(E-24 で具体化、v2.0)

#### 7.3.1 基本方針(Phase 5 確定)

- 個人の正解率で **自動調整**
- 教師が **「上限・下限」の枠** を設定
- ユーザがいつでも **上下調整可**(イニシアチブ原則)
- 児童・生徒に表示する文言: **感情ベース**(「もっと頑張りたい」「ゆっくりやりたい」)
- 数値・段階名(「ふつう」「むずかしい」)は合理的配慮で見せない方向で実装

#### 7.3.2 判定アルゴリズム: 移動平均型(E-24 確定)

- **直近5問の正解率** で判定
  - 5問中 4問以上正解 → **1段階上昇**(上限あり)
  - 5問中 2問以下正解 → **1段階下降**(下限あり)
  - 5問中 3問正解 → 現状維持
- **3段階**: やさしい / ふつう / むずかしい
- **教科別独立管理**: 算数得意・国語苦手 等の個別最適化
- イベント駅・カード使用は判定対象外(クイズ正答率のみで判定)

#### 7.3.3 教師の枠設定

- 「上限=ふつう」「下限=やさしい」のように枠を狭めて児童のレベル変動を制御
- 学級全体の上限・下限、または個別児童の上限・下限を設定可
- デフォルト: 上限「むずかしい」、下限「やさしい」(全範囲)

#### 7.3.4 ユーザ手動調整との関係

- ユーザが「もっと頑張りたい」(上)または「ゆっくりやりたい」(下)を押した場合:
  - **即座に1段階変動**
  - 自動判定は **5問休止** → 5問後に再開
- 手動意思を尊重しつつ、自動判定を恒久停止はしない(イニシアチブ原則と動的最適化のバランス)

#### 7.3.5 教師向け説明

教師ダッシュボードで以下を可視化:
- 各児童・生徒の現在の難易度(教科別)
- 直近5問の正解状況
- 過去の難易度変動履歴(グラフ)

### 7.5 ヒントの仕様

#### 7.5.1 ヒントの中身: 視覚+関連語ハイブリッド

絵 + 短い関連語句の組合せ。 **1段階で完結**(段階的でない)。

例: 「東京の名物は?」
- ヒント: 🍳(鉄板の絵) + 「とうきょうの したまちで うまれた おりょうり」

#### 7.5.2 ヒント表示タイミング

| タイミング | きっかけ |
|---|---|
| 不正解時 | 自動表示 |
| 事前 | 💡 ヒントカード使用 |

### 7.6 不正解時の処理: 学習接続版

#### 7.6.1 流れ

```
不正解
  ↓
ヒント表示(視覚+関連語)
  ↓
🔁 もう一度挑戦  ← もう一度挑戦カード持ってる時だけ表示
  │
  ├─ カード使用 → 再挑戦 → 正解で物件・コインゲット
  │
  └─ カード未使用 or 不正解
       ↓
       不正解問題を 学習DBに記録
       ↓
       次のプレイヤーへ
       ↓
[次回プレイ時] 「ふくしゅう」として類題出題
```

#### 7.6.2 学習接続データ

- 児童・生徒個人ごとに「苦手問題リスト」を保持
- 動的難易度調整に反映
- 教師ダッシュボードで可視化

### 7.7 教育ダッシュボード(必須機能)

#### 7.7.1 教師向けダッシュボード

- 児童・生徒ごとの **正解率推移**
- **苦手問題リスト**(類題出題対象)
- 動的難易度の **現在状態**
- **プレイ履歴**(セッション・年度・スコア)
- 教科別・単元別の **習熟度マップ**

#### 7.7.2 児童・生徒向け(イニシアチブ原則)

- **「ふくしゅうリスト」**(自分の苦手問題、覚え直し用)
- 自分の正解数推移(肯定的に表示)
- 自分の難易度状態(感情ベース表示で)

#### 7.7.3 技術要件への影響

教育ダッシュボードを必須機能とすることで、Phase 6で以下が必須に:
- 認証システム(児童・生徒の識別)

## 8. 全UI方針: 漢字+ルビ(Phase 2 確定、v1.9 で Q4-1-7/Q4-1-8 確定)

### 8.1 表記方針(Q4-1-7 確定)

**A案: 常用漢字+ルビ、例外あり** を採用。

- 常用漢字を基本に表示、学年配当外は全ルビ
- 駆動語彙(「鉄道」「駅」など固有頻出語)は学年配当外でも漢字+ルビで提示し定着促進
- **すべての文言を「漢字+ルビ」で統一**(駅名・カード名・物件名・キャラ名も含む)
- 例外: カタカナ表記が自然な駅名(「ピカピカシティ駅」等)はそのまま

### 8.2 アクセシビリティ設定(Q4-1-7 確定)

- **ルビON/OFF切替** が UIボタンで可能
- 設定の場所:
  - **ゲーム開始前のモード選択画面**
  - **ゲーム中の歯車アイコン(常時アクセス可)**
  - **教師アカウントから一括設定**(クラス全員に適用)
- 初期値: **C層モード=ON(ルビあり)**
- 駅名・物件名・カード名・キャラ名すべてに連動

### 8.3 表示比較

```
[ON: 漢字+ルビ]                  [OFF: 純ひらがな]
┌─────────────────┐             ┌─────────────────┐
│   みかん  ばたけ  │             │                 │
│   蜜柑  畑  駅    │  ⇄         │  みかんばたけ駅   │
│                  │             │                 │
│   か              │             │                 │
│   買う?           │             │  かう?           │
└─────────────────┘             └─────────────────┘
```

設計判断: OFF は「ルビなしの漢字表示」ではなく **「純ひらがな表示」** とする。C層配慮として、漢字が読めない学習者でもプレイ可能にするため。読字困難層への合理的配慮の一環。

### 8.4 データ層の設計(技術メモ、Q4-1-8 確定)

```json
{
  "id": "station_001",
  "name": {
    "kanji": "蜜柑畑駅",
    "hiragana": "みかんばたけえき",
    "ruby": [["蜜柑", "みかん"], ["畑", "ばたけ"]]
  }
}
```

- HTML5 `<ruby>` タグで実装
- グローバル設定で `data-ruby="on|off"` を切替
- **架空駅名・物件名・カード名・キャラ名・問題文・UIテキストすべてに同方針を適用**(Q4-1-8 確定)
- すべてのテキストデータを kanji形・hiragana形・ruby情報の3形式で保持
- カタカナ駅名(ピカピカシティ駅等)は ruby 配列が空で、kanji = hiragana となる

## 9. 合理的配慮原則(9歳の壁対策 + イニシアチブはユーザ)

第1弾(C層)の核心。すべてのUI/UX判断はこの3原則に従う。

### 9.1 抽象→具体

| 抽象表現(NG) | 具体表現(OK) |
|---|---|
| 「100万円」 | コイン100枚の絵 |
| 「収益率5%」 | 「10こ おいたら はんぶん もらえる」+ 絵 |
| 「決算」 | コインがたくさん転がってくる演出 |
| 「総資産1位」 | 一番たくさん コイン・たてもの を持っている人 |

### 9.2 行間ゼロ

- 比喩・暗喩を排除
- 敬語を排除(「ご購入されますか?」→「**かう?**」)
- 慣用句を排除(「目から鱗」「足元をすくわれる」など)
- すべて字面通りの表現

### 9.3 ビジュアライズ強化

- すべての操作・状態変化を **アニメーションで可視化**
  - 物件購入 → 建物が建つアニメ
  - 決算 → コインが転がる演出
  - ターン交代 → 物理的バトンの受け渡し演出
  - サイコロ → 大きく振動するアニメ
- 文字情報より **動きと絵** を優先

### 9.4 セーブ・再開機能(必須)

- 余暇モード(20〜30年)を一回で終えるのは現実的でない
- 任意のターンで中断可能
- 中断状態を保存し、後日続きから再開
- 教室利用時は授業のチャイムで強制中断対応

### 9.5 イニシアチブはユーザ(Phase 5 で追加された原則)

すべてのインタラクションで **児童・生徒(=ユーザ)が主導権を持つ**。教師は監督・必要時介入・枠の設定者。

| 領域 | ユーザのイニシアチブ |
|---|---|
| 動的難易度 | 自動調整 + 自分で上下調整可 |
| 制限時間 | 「ゆっくり/ふつう」を選択 |
| ヒント・たすけ | 自分から「ほしい」と能動的に求める |
| もう一度挑戦 | 自分で再挑戦を選ぶ |
| 学習接続 | 自分の「ふくしゅうリスト」が見える |
| 教師の役割 | 監督・上限下限設定・必要時介入のみ |

## 10. 授業形態と技術構成

### 10.1 対応する3つのプレイ形態

| 形態 | 想定シーン |
|---|---|
| **A) 教室1台ホットシート** | 1台のPCを順番に操作、他の児童・生徒は画面を見て学ぶ |
| **B) 教師PC + 生徒タブレット** | 教師が司会、生徒は手元のタブレットでも操作可 |
| **C) オンライン同期** | 各自家から同じセッションに参加、画面共有的に進行 |

すべての形態で「見て学ぶ」(同期観察)が成立する。

### 10.2 共通基盤の考え方

```
┌─────────────────────────────────────────────────┐
│ 共通基盤: セッション同期エンジン                    │
│ (1ゲーム = 1ルーム、状態はサーバー集中管理)        │
└──────────────┬──────────────────────────────────┘
               │
        ┌──────┼──────┬─────────────┐
        ▼      ▼      ▼
  A: 1クライアント   B: ホスト+N子機   C: 各自独立クライアント
```

技術的には **B) ホスト+子機構造を共通基盤** とし、A) は「子機0台のB)」、C) は「子機が遠隔のB)」として実現する。

詳細な技術スタック選定は **Phase 6** で行う。

### 11.4 リアルタイム同期: Supabase Realtime

3形態すべて同じコードで動作:


## 13. 教科横断の取り扱い

第1弾で取り扱う教科:

- **地理**: 実在地名(駅副表示)・地域産業・気候・地形
- **経済**: コイン・物件・決算を通じた基礎概念
- **公民**: 市場のしくみ・税・社会制度
- **情報**: データ・AIキャラとの対話・可視化リテラシー
- **家庭**: 食材産地・調理・生活設計
- **数学**: 数の比較・図形・確率(サイコロ)・基礎計算

教科横断モードでは、1試合の中で **複数教科のクイズが混在出題** される。

## 15. API 設計(Phase 7 進行中、v2.2)

### 15.1 API 全体構造: 3層 + Realtime

Phase 6 で確定した「Supabase オールインワン」と整合する API スタイル(Phase 7 v2.2 で確定)。

| Layer | 実装場所 | 用途 | 認可 |
|---|---|---|---|
| **Layer 1** | Supabase Client SDK(クライアント直接) | シンプルな CRUD、Realtime 購読 | RLS で宣言的に制御 |
| **Layer 2** | Next.js Server Action | ゲームロジック、状態遷移、計算 | Server Action 内で auth.uid() 確認 + DB 操作 |
| **Layer 3** | Edge Function(Vercel Functions) | LLM 呼出、マップ生成、バッチ処理 | Function 内で署名検証 + Service Role |

Realtime は **Layer 1 から直接購読**(Supabase Realtime channel)、書込は **Layer 2 Server Action 経由**(不正防止のため)。

### 15.2 Layer 1: Supabase Client SDK 主要操作

クライアント側 TypeScript で直接呼ぶシンプルな CRUD と Realtime 購読。

```ts
// 例: 自分のプロファイル取得
const { data, error } = await supabase
  .from('profiles')
  .select('*')
  .eq('user_id', user.id)
  .single();

// 例: 進行中ゲーム購読
const channel = supabase
  .channel(`game_session:${sessionId}`)
  .on('postgres_changes',
      { event: 'UPDATE', schema: 'public', table: 'game_sessions', filter: `id=eq.${sessionId}` },
      (payload) => updateUI(payload.new))
  .subscribe();
```

主要操作カテゴリ(Phase 7-ER 21テーブルに対応):

| カテゴリ | テーブル | 操作 |
|---|---|---|
| 認証 | (Supabase Auth) | signup / signin / signout / passwordReset |
| プロファイル | `profiles` | 自分の取得・更新、ニックネーム任意(15.6 厳格化版) |
| クラス | `classes`, `class_memberships` | 自分の所属・担当クラス取得 |
| ゲームセッション履歴 | `game_sessions`, `session_participants` | 自分の参加履歴 |
| クイズ出題プール | `quizzes`, `quiz_pools` | 教科×単元×難易度でフィルタ取得 |
| 物件マスタ | `properties`, `property_stations` | 駅別物件マスタの読取 |
| 自分の所有物件 | `owned_properties` | 自分の所有物件取得 |
| 自分のカード | `owned_cards` | 自分の保有カード取得(最大3枚、E-22) |
| 設定 | `accessibility_settings`, `difficulty_states` | 自分の設定取得・更新(Layer 1で可)、難易度状態は読取のみ |
| 学習進捗 | `quiz_attempts`, `review_lists` | 自分のふくしゅうリスト、正解数推移 |
| 教師ダッシュボード | (上記の集計ビュー) | 教師ロールが自クラスのみ読取(RLS) |
| Realtime | `game_sessions` 他 | session_id 単位で購読 |

### 15.3 Layer 2: Server Action 詳細(10個)

ゲームロジックの腰、不正防止のためサーバ側で必ず実行。すべて `app/actions/game.ts` 等に配置。

#### 15.3.1 `gameSession.start(input)`

```ts
type Input = {
  classId: string;
  participantUserIds: string[];          // ホットシート: [user1, user2,...] / 単独: [user1]
  mode: 'icebreak' | 'leisure';
  unitFilter?: { subjectIds: string[]; unitIds: string[] };  // 単元連動オプション(E-25 D)
  mapSeed?: string;                       // シード固定(E-25)
};
type Output = { sessionId: string; mapId: string };
```

処理フロー:
1. `auth.uid()` がクラスに所属することを確認
2. Edge Function `generate-map` を呼出(E-25 アルゴリズム適用)
3. `game_sessions` 行を INSERT(status='in_progress', current_turn=1, current_player_index=0)
4. `session_participants` を INSERT(参加者全員、初期化済の coin/score)
5. 物件マスタからこのマップ用の物件をシード → `session_properties`
6. クイズプールから出題候補をシード → `session_quiz_pool`
7. 開始イベントを Realtime channel に publish

権限: student/teacher 両方可、teacher の場合は授業開始扱い、student の場合は自由プレイ扱い。

エラー: マップ生成失敗 / RLS 違反 / 参加者がクラスに居ない。

#### 15.3.2 `gameSession.rollDice(input)`

```ts
type Input = { sessionId: string };
type Output = {
  diceValue: 1|2|3|4|5|6;
  newStationId: string;
  passedStations: string[];   // 通過駅のID(コイン獲得)
  arrivedEvent: 'normal' | 'event_station' | 'goal';
};
```

処理フロー:
1. session 取得、`current_player_user_id` が `auth.uid()` と一致するか確認
2. 災難キャラの影響を計算(目半分/休み等)
3. サーバ側で乱数生成(`crypto.randomInt(1, 7)`、不正防止)
4. マップから到着駅を計算(円環+分岐、E-25 アルゴリズム参照)
5. 分岐に到達した場合は `pending_branch_choice` を立てて呼出元に返す(別 Action で確定)
6. 通過駅ごとに +1〜2 コイン
7. `game_sessions.current_station_id` を UPDATE
8. Realtime に publish

権限: 自分のターンの時のみ可。

エラー: 自分のターンではない / セッション終了済 / 災難キャラで休みターン。

#### 15.3.3 `gameSession.submitAnswer(input)`

```ts
type Input = {
  sessionId: string;
  quizId: string;
  selectedChoice: 'A'|'B'|'C';
  elimSelections?: ('A'|'B'|'C')[];     // 消去法で × にした選択肢(操作履歴)
  hintUsed: boolean;
  timeUsedSec: number;
};
type Output = {
  correct: boolean;
  coinDelta: number;
  newProperty?: { propertyId: string; name: string };
  hint?: { image: string; text: string };   // 不正解時
  retryAvailable: boolean;                  // もう一度挑戦カード所持時
  newDifficultyLevel?: 'easy' | 'normal' | 'hard';  // E-24 判定で変動した場合
};
```

処理フロー:
1. session 状態を取得、自分のターンか確認
2. クイズの正解を `quizzes` から取得
3. 正解判定
4. `quiz_attempts` を INSERT(正誤、消去法操作、ヒント使用、所要時間)
5. **正解時**:
   - コイン +N(駅・難易度ベース)
   - 駅の物件を `owned_properties` に追加 → 独占判定 → ボーナス算定
   - 不正解履歴があれば `review_lists` から削除
6. **不正解時**:
   - ヒント生成(視覚+関連語、Phase 5 確定)
   - `review_lists` に追加(ふくしゅう対象、7.6 確定)
7. **動的難易度判定**(E-24):
   - 直近5問の正解率を算出(`quiz_attempts` 集計)
   - 4/5 以上 → 上昇、2/5 以下 → 下降、教師の枠内に収める
   - `difficulty_states` を UPDATE(教科別)
8. Realtime に publish

権限: 自分のターンの時のみ可、本人の quiz_attempts のみ書込。

エラー: 不正なクイズID / 既に回答済み / セッション終了済。

#### 15.3.6 `gameSession.useCard(input)`

```ts
type Input = {
  sessionId: string;
  cardId: string;
  cardType: string;                    // 'speed'|'warp'|'shield'|...
  targetStationId?: string;            // ワープ系で必要
  targetQuizId?: string;               // ヒント系で必要
};
type Output = { effect: object; remainingCards: number };
```

処理フロー:
1. カードを所有しているか確認(`owned_cards`)
2. カード種別ごとに効果を適用:
   - 🚄 すばやさカード → 次の rollDice で 2d6 になる(`game_sessions.next_dice_count=2`)
   - 🚅 ひとっとびカード → `current_station_id` を target に書換、移動演出
   - 🛡️ まもりカード → 次のマイナスイベントを 1回無効化(`pending_shield=true`)
   - 🔄 ふりなおしカード → `rollDice` を即座にもう一度
   - ↩️ ひきかえしカード → 1ターン前の `current_station_id` に戻す
   - ⏪ まきもどしカード → セッションスナップショット復元(`session_snapshots`)
   - 🎯 しゅうてんカード → 目的地駅にワープ
   - 💡 ヒントカード → 次のクイズで自動ヒント表示
   - 🤝 おてつだいカード → AIキャラが回答候補を絞る(LLM呼出)
   - 🔁 もう一度挑戦カード → 不正解後の再挑戦を可能化
3. `owned_cards` から DELETE
4. Realtime に publish

権限: 自分のターン、カード所有時のみ。

エラー: カード未所有 / カード種別不正 / 対象不正(ワープ先など)。

#### 15.3.7 `gameSession.acquireCard(input)`(E-22/E-26 確定)

```ts
type Input = { sessionId: string; source: 'card_station' | 'quiz_correct' | 'present_event' };
type Output =
  | { acquired: true; cardId: string; cardType: string }
  | { acquired: false; reason: 'limit_reached'; choice: 'discard' | 'skip' };
```

処理フロー:
1. **入手抽選**(E-26):
   - source='card_station' → 100% でランダム1枚
   - source='quiz_correct' → 15% でランダム1枚
   - source='present_event' → 50% でランダム1枚(残り 50% は物件)
2. **上限判定**(E-22): 現在のカード保有数 = 3 なら、選択モーダル要請を返す
3. 選択モーダル後、再呼出で `choice` を受ける:
   - choice='discard' → どのカードを捨てるかは別 Action(`discardCard`)で先に処理 → 再 acquireCard
   - choice='skip' → 入手スキップ
4. 確定時 `owned_cards` に INSERT
5. Realtime に publish

権限: 自分のターン、source 妥当性検証。

エラー: source 不正 / セッション終了済。

#### 15.3.8 `gameSession.endTurn(input)`

処理フロー:


処理フロー:
1. 全参加者について、所有物件の決算時収益を計算(物件価格 × 1/5、5.2.3)
2. 訪問賃料の集計(セッション中に蓄積したもの、物件価格 × 1/20)
3. `session_participants.coin` を UPDATE
4. 年度結果を `settlement_logs` に保存
5. アイスブレイクモード(1〜3年)なら最終年度判定
6. 余暇モード(20〜30年)なら継続
7. 最終年度なら `gameSession.end()` 内部呼出 → スコア計算(3部門制、5.3)
8. 演出フラグと Realtime publish

権限: 自動呼出のみ、外部からは呼べない(security definer)。

エラー: 既に決算済 / 計算不整合(整合性チェック)。

### 15.7 エラーハンドリング方針

- **Layer 1(Supabase Client)**: SDK の error オブジェクトをそのままクライアントで処理(`{ data, error } = await supabase...`)
- **Layer 2(Server Action)**: `try/catch` + 構造化エラー型を返却。次の3カテゴリで分類:
  - `ValidationError`(入力不正、ユーザに表示)
  - `AuthorizationError`(権限不足、ログイン誘導)
  - `BusinessLogicError`(コイン不足等、ユーザに表示)
  - `SystemError`(DB障害等、エラー画面)
- **Layer 3(Edge Function)**: 同上 + Sentry/Vercel Logs に送信、ユーザには「処理に失敗しました」の汎用メッセージ
- **クライアント側 UX**: C層配慮として、エラーメッセージは絵文字+ルビ付き+「もう一度やってみる」ボタンで再試行誘導

### 15.8 RLS ポリシー詳細(21テーブル × 3ロール、Phase 7-RLS 確定、v2.4)

12.3 のER図を入力に、21テーブル全てに対する RLS ポリシーを **Supabase SQL** で確定。実装時はそのまま migration に貼り付けて使用可能。

#### 15.8.1 全体方針

- **RLS は全テーブルで `ENABLE ROW LEVEL SECURITY` 必須**(Supabase の安全側初期値)
- **public ロールには何も grant しない**(authenticated/service_role のみ操作)
- **3ロールを JWT カスタムクレーム `role` で識別**(Supabase Auth の `app_metadata.role` を JWT に含める)
- **書き込み系の重要操作は Server Action 経由**(Layer 2、15.3)、RLS は二段目防御として機能
- **`USING` 句は読取/更新/削除の対象行を絞る**、**`WITH CHECK` 句は INSERT/UPDATE 後の行を検証**
- **教師の "自クラスのみ" は ヘルパー関数 `is_teacher_of(class_id)` で集約**(ポリシーが冗長化しないため)

#### 15.8.2 ヘルパー関数とビュー

```sql
-- ============================================================
-- JWT クレーム取得
-- ============================================================
create or replace function auth.user_role() returns text
language sql stable as $$
  select coalesce(
    auth.jwt() -> 'app_metadata' ->> 'role',
    'student'
  );
$$;

-- ============================================================
-- 担当クラス判定(教師ロール用)
-- ============================================================
create or replace function public.is_teacher_of(target_class_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select exists(
    select 1 from classes c
    where c.id = target_class_id
      and c.teacher_user_id = auth.uid()
  ) or exists(
    select 1 from class_memberships m
    where m.class_id = target_class_id
      and m.user_id = auth.uid()
      and m.role = 'teacher'
  );
$$;

-- ============================================================
-- 同セッション参加者判定(児童同士はゲーム情報を相互参照可)
-- ============================================================
create or replace function public.is_in_session(target_session_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select exists(
    select 1 from session_participants p
    where p.session_id = target_session_id
      and p.user_id = auth.uid()
  );
$$;

-- ============================================================
-- 自クラスのセッション判定(教師)
-- ============================================================
create or replace function public.session_in_my_class(target_session_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select exists(
    select 1 from game_sessions g
    where g.id = target_session_id
      and is_teacher_of(g.class_id)
  );
$$;

-- ============================================================
-- 自クラスの児童判定(教師ダッシュボード用)
-- ============================================================
create or replace function public.is_my_student(target_user_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select exists(
    select 1 from class_memberships m
    join classes c on c.id = m.class_id
    where m.user_id = target_user_id
      and m.role = 'student'
      and (c.teacher_user_id = auth.uid()
           or exists(select 1 from class_memberships tm
                     where tm.class_id = c.id
                       and tm.user_id = auth.uid()
                       and tm.role = 'teacher'))
  );
$$;

-- ============================================================
-- admin 判定
-- ============================================================
create or replace function public.is_admin() returns boolean
language sql stable as $$
  select auth.user_role() = 'admin';
$$;
```

#### 15.8.5 カラムレベル GRANT(教師の枠設定など)

RLS では行レベルしか制御できないため、**特定カラムだけ更新可** という要件は GRANT で補強する。

```sql
-- 教師は difficulty_states の bounds 系のみ更新可
revoke update on difficulty_states from authenticated;
grant update (lower_bound, upper_bound) on difficulty_states to authenticated;
-- (current_level, recent_correct_count は service_role のみ)

-- 児童は profiles の role を更新できない(15.8.4 で WITH CHECK でも担保)
revoke update on profiles from authenticated;
grant update (nickname, region_code) on profiles to authenticated;
```

#### 15.8.6 Realtime と RLS の整合(15.5 補足)

- Realtime チャネル `game_session:{sessionId}` は **postgres_changes** で `game_sessions` テーブルの UPDATE を購読する
- Supabase Realtime は **RLS を尊重** するため、`is_in_session(id)` で false になる利用者には変更通知が届かない
- 教師が観戦するときは `session_in_my_class(id)` で透過的に通知を受信できる
- 設計上、不正に他クラスのセッションを購読してもイベントが流れない（防御の二段目）

#### 15.8.7 Server Action と service_role の使い分け

| 操作 | 経路 | 理由 |
|---|---|---|
| プロフィール表示名変更 | Layer 1(Client SDK + RLS) | RLS で十分 |
| サイコロ・移動・回答 | Layer 2(Server Action + service_role) | 改竄防止 |
| カード入手・使用 | Layer 2(Server Action + service_role) | E-22 上限・E-26 確率を server で確定 |
| マップ生成 | Layer 3(Edge Function + service_role) | 重い計算 + シード固定 |
| 同意取得・撤回 | Layer 2(Server Action + service_role) | 暗号化処理が必要 |
| 教師の難易度枠設定 | Layer 1(Client SDK + RLS + カラムGRANT) | RLS + GRANT で十分 |

#### 15.8.8 テスト方針(F-27 と整合)

RLS は **PostgreSQL ロール切替テスト** で検証する(`set local role authenticated; set local request.jwt.claims = '...'::jsonb;`)。

| テストケース | 期待挙動 |
|---|---|
| 児童Aが児童Bの quiz_attempts を SELECT | 0行 |
| 教師が他クラス児童の review_lists を SELECT | 0行 |
| 児童が自身の profiles.role を 'admin' に UPDATE | エラー(WITH CHECK 失敗) |
| 児童が parental_consents を SELECT | 0行 |
| 教師が difficulty_states.current_level を直接 UPDATE | エラー(GRANT なし) |
| service_role が全テーブルに CRUD | 成功(RLS バイパス) |

実装時は `pgTAP` または **Supabase CLI の seed + テスト** で全21テーブル × 3ロール × CRUD の網羅検証。

### 16.4 状態管理戦略

#### 16.4.1 4つのストアを Zustand で分離

| Store | スコープ | 永続化 |
|---|---|---|
| `userStore` | 認証情報、ロール、自分のプロファイル | Supabase Auth 同期 |
| `accessibilityStore` | アクセシビリティ設定(ルビ・サイズ・音・%表現等) | Supabase + localStorage |
| `gameStore` | 進行中ゲームのローカル状態(楽観的更新) | session 中のみ |
| `uiStore` | モーダル開閉・トースト等の一時UI状態 | なし |

#### 16.4.2 Realtime 同期パターン

```tsx
// useGameSession.ts
function useGameSession(sessionId: string) {
  // 1. 初期取得: Supabase Client
  const { data: session } = useSWR(`session-${sessionId}`, fetchSession);
  
  // 2. Realtime 購読: postgres_changes チャネル
  useEffect(() => {
    const channel = supabase
      .channel(`game_session:${sessionId}`)
      .on('postgres_changes', { ... }, (payload) => {
        gameStore.merge(payload.new);
      })
      .subscribe();
    return () => channel.unsubscribe();
  }, [sessionId]);
  
  // 3. ローカル楽観的更新: Server Action 呼出時
  return { session: gameStore.session, ...actions };
}
```

#### 16.4.3 Server Action 呼出パターン

```tsx
// クライアントコンポーネント
'use client';
async function handleDiceClick() {
  uiStore.setLoading(true);
  try {
    const result = await gameSessionActions.rollDice({ sessionId });
    gameStore.applyDiceResult(result);
  } catch (e) {
    if (e instanceof BusinessLogicError) {
      uiStore.showToast(e.message);
    } else if (e instanceof AuthorizationError) {
      router.push('/login');
    } else {
      uiStore.showErrorModal();
    }
  } finally {
    uiStore.setLoading(false);
  }
}
```

#### 16.5.2 `<GameBoardCanvas />` の実装パターン

```tsx
'use client';
import { Application } from 'pixi.js';
import { useRef, useEffect } from 'react';

function GameBoardCanvas({ map, players, currentStationId }) {
  const ref = useRef<HTMLDivElement>(null);
  const appRef = useRef<Application>();
  
  // Pixi Application を一度だけ初期化
  useEffect(() => {
    const app = new Application({ resizeTo: ref.current });
    appRef.current = app;
    ref.current!.appendChild(app.view as HTMLCanvasElement);
    return () => app.destroy(true);
  }, []);
  
  // map/players の変更を Pixi に反映
  useEffect(() => {
    drawMap(appRef.current!, map);
  }, [map]);
  
  useEffect(() => {
    animatePlayerMove(appRef.current!, currentStationId);
  }, [currentStationId]);
  
  return <div ref={ref} className="w-full h-full" />;
}
```

#### 16.6.3 タイポグラフィ最終確定(v2.5)

##### 基本フォント

| 役割 | フォント | 理由 |
|---|---|---|
| 本文・UI | **Noto Sans JP** | Google Fonts 提供、CJK 全カバー、商用無料、可読性高 |
| ルビ | Noto Sans JP(本文と同フォント) | フォント統一で読みやすさ最大化([[feedback_ruby_dict_conflicts]] と整合) |
| ピクセルアートUI(ゲーム内) | **DotGothic16** | レトロドット風、Noto と並存可能、「ゲーム」と「UI」を視覚的分離 |
| 数字(コイン・スコア) | Noto Sans JP の `font-feature-settings: "tnum"` | 等幅化で揃える |

##### サイズスケール(Tailwind 拡張)

C層配慮で **既定が大きめ**。font-scale 1x/1.5x/2x で追従(`accessibility_settings.font_scale`、12.3)。

| Tailwind | px(1x) | 用途 |
|---|---|---|
| `text-xs` | 14px | ルビ専用(本文の約 60%) |
| `text-sm` | 16px | 補助・タグ |
| `text-base` | 18px | **既定本文**(通常16より大きめ) |
| `text-lg` | 22px | リード文・カード見出し |
| `text-xl` | 28px | 画面見出し |
| `text-2xl` | 36px | クイズ問題文・大見出し |
| `text-3xl` | 48px | コイン表示・ターン数等の主要KPI |
| `text-4xl` | 64px | ロゴ・「正解！」モーダル |

##### line-height(行間)

9.2「行間ゼロ原則」と整合し、**過剰な行間を排除**:

| サイズ範囲 | line-height | 備考 |
|---|---|---|
| ルビ(text-xs) | `1` | 本文と密着 |
| 本文(text-base〜text-lg) | `1.5` | 詰めすぎず読みやすい標準 |
| 見出し(text-xl 以上) | `1.2` | コンパクトで強い印象 |
| 単行UI(ボタン・タグ) | `1` | 高さ計算が容易 |

「行間ゼロ」は **段落間の縦余白を最小化** の意であり、line-height 1.0 を強制するわけではない。読みやすさを損なわない範囲で詰める。

##### font-feature-settings

```css
body {
  font-family: 'Noto Sans JP', sans-serif;
  font-feature-settings:
    "palt"  1,  /* プロポーショナル詰め(漢字+カナの空間最適化) */
    "kern"  1,  /* カーニング */
    "tnum"  1;  /* 等幅数字 */
  font-variant-numeric: tabular-nums;  /* 数字揃え(score 表示で重要) */
}

ruby rt {
  font-feature-settings: "kern" 1;
  /* ルビは palt しない(揺らぎ防止) */
}
```

##### ルビと漢字の距離([[feedback_ruby_spacing]])

| プラットフォーム | スタイル | ノウハウ |
|---|---|---|
| Web(Next.js) | CSS `ruby-position: over;` + `rt { line-height: 1; margin-bottom: 1px; }` | line-heightを1に詰めて縦距離を最小化 |
| iOS(将来 Capacitor 化) | `marginTop: -2` | 既存モバイル版の知見を継承 |

## 18. Phase 7.5 確定: 法務・ビジネス・法令整合

第三次審査で発覚した35項目のうち、A系(法務)とB系(ビジネス)、および学校教育法・障害者差別解消法との整合検証を Phase 7.5 として整理した。

### 18.1 法務リスク回避(v1.7 で先行確定)

商標法・著作権法・不正競争防止法のリスク評価を実施。詳細は `legal-analysis.md` 参照。

| 法分野 | リスク評価 | 対応状況 |
|---|---|---|
| 商標法 | 低(完全独自名称) | ✅ マーケ・ゲーム内・ドキュメント全般から特定先行作品名を排除済 |
| 著作権法 | 低(独自表現、メカニクス共通は保護対象外) | ✅ 独自キャラ・固有名称・ストーリーで構成 |
| 不正競争防止法 | 低(マーケ表記排除済) | ✅ 「インスパイア」表記の中立化 |

特定先行作品名・キャラクター名・カード名を独自表現に置換済み(v1.7)。CPU対戦キャラは花テーマで再命名(サクラ / ヒマワリ / モミジ / ユキ)。リリース前の弁理士相談で最終確認予定。

### 18.2 課金モデル(B-8 確定)

**3層構造のフリーミアム + B2B**(具体価格はパイロット後決定):

| 層 | 構成 | 想定アクセス経路 |
|---|---|---|
| **Free(個人)** | 1教科(地理) / 5駅 + イベント駅5種 = 10駅 / ソロ+ホットシート2人 / 合理的配慮全機能利用可 / 児童データはローカルのみ | 検索・SNS・口コミ |
| **プレミアム(個人)** | 全教科解放 / 全駅マップ / オンライン同期マルチ / 親モード / Supabase 同期 | アプリ内決済 |
| **教育機関ライセンス** | フル機能 + クラス管理 + 教師ダッシュボード + 学年集計、生徒数比例の年額契約 | 学校・自治体直接契約 |

#### 設計原則

- 合理的配慮機能は Free 層でも全機能利用可(教育的公平性の観点)
- C層家庭の支払能力を考慮し、個人プレミアムは低価格帯を意図
- LLM ランタイムコストが低い設計(問題生成・キャラセリフは事前バッチ)のため、無料層を厚くしても破綻しない

#### 価格決定の保留事項

- 個人プレミアムの月額・買い切り価格(月数百円〜千円帯で検討中)
- 教育機関ライセンスの生徒単価(年数百円帯で検討中)
- 自治体一括契約の単価(個別見積)

→ パイロットテスト(Phase 9)後に実測ベースで決定。

5. **法務クリア独自IP**: 桃鉄教育版に対して家庭・自治体導入両面で展開可能

### 18.7 COPPA / GDPR-K 判断(A-7 確定)

#### 適用判断

| 法令 | 適用 | 対応方針 |
|---|---|---|
| 日本 個人情報保護法 | 第1弾から適用 | 法定代理人(保護者)同意で対応 |
| 米国 COPPA(13歳未満) | 国際展開時のみ | 設計に「年齢区分・地域別同意フロー」識別子の余白を残す。第1弾では非対応 |
| EU GDPR-K(16歳未満、国により13歳まで引下げ可) | 国際展開時のみ | 同上 |
| 学校教育法・自治体個人情報保護条例 | 学校導入時に適用 | 学校契約書で要件明示 |

#### 設計余白として残すフラグ

| フラグ | 用途 |
|---|---|
| `region_code` (JP/US/EU/...) | 地域別同意フロー切替 |
| `consent_scope` (parental / institutional) | 個人・学校・放デイの同意経路区別 |
| `consent_version` | 規約更新時の再同意トリガー |

→ 第1弾は日本国内法のみ準拠。国際展開は将来対応(余白として識別子を設計)。

### 19.4 評価ルーブリック設計(C-16 確定)

#### 19.4.1 基本方針

学習指導要領の **3観点評価**(2017/2018年改訂で全教科共通) に整合:
- ① 知識・技能
- ② 思考力・判断力・表現力
- ③ 主体的に学習に取り組む態度

各観点を **A(十分達成) / B(おおむね達成) / C(支援が必要)** の3段階で評価。tetsu-board は **デジタル指標(自動計測)+教師観察(手動)+児童自己評価(主観)** の3面から判定。

#### 19.4.2 観点①「知識・技能」ルーブリック

| 段階 | 達成水準 | デジタル指標(自動) | 教師観察項目 |
|---|---|---|---|
| A | 単元の中心概念を確実に理解、応用問題も解ける | クイズ正答率 80% 以上、ヒント使用率 30% 未満 | 「なぜそうなる?」を自分の言葉で説明できる |
| B | 単元の基本概念を理解、典型問題は解ける | クイズ正答率 60〜79%、ヒント使用率 30〜60% | 教師の補助で説明できる |
| C | 単元の基本概念に支援が必要 | クイズ正答率 60% 未満、ヒント使用率 60% 以上 | 視覚的補助・具体物が必要 |

#### 19.4.3 観点②「思考力・判断力・表現力」ルーブリック

| 段階 | 達成水準 | デジタル指標(自動) | 教師観察項目 |
|---|---|---|---|
| A | 複数情報を統合し最適解を選択 | カード使用判断のタイミング適切、消去法UI(7.4.2)を戦略的に使用 | 「自分はこう考えた」を理由付きで言える |
| B | 与えられた選択肢から妥当な判断 | 物件購入時に駅カテゴリを意識、無謀な動きが減る | 教師との対話で理由を整理できる |
| C | 判断の理由づけに支援が必要 | ランダムな購入・カード使用、戦略性が見られない | 教師の問いに対して「わからない」が中心 |

#### 19.4.4 観点③「主体的に学習に取り組む態度」ルーブリック

| 段階 | 達成水準 | デジタル指標(自動) | 教師観察項目 |
|---|---|---|---|
| A | 自ら学びを深め、ふくしゅうにも取り組む | ふくしゅうリスト消化率 70% 以上、設定変更で自分に合う環境を作る | 「もっとやりたい」「次は○○を学びたい」と発言 |
| B | 授業時間中は集中して取り組む | ふくしゅうリスト消化率 30〜69%、教師指示の設定で取り組む | 教師の声かけで継続 |
| C | 取り組みに継続的支援が必要 | ふくしゅうリスト未消化、設定に無関心 | 注意散漫、課題回避傾向 |

#### 19.4.5 児童・生徒 自己評価項目(WF6 / 14.6 と接続)

ゲーム終了時に画面に表示される簡易自己評価。`accessibility_settings` 経由でON/OFF可。

| 質問 | 回答方式 |
|---|---|
| 「きょうはたのしかった?」 | 絵文字3択(笑顔/普通/つらい) |
| 「もっとやってみたい?」 | はい/いいえ/どちらでも |
| 「むずかしかったところはあった?」 | はい+カテゴリ選択/いいえ |
| 「つぎは なに を まなびたい?」 | 教科アイコン6個から選択(任意) |

回答は `quiz_attempts` 系とは別の `self_evaluations` テーブル(後日追加候補、12.3 余白)へ記録。

#### 19.4.6 教師ダッシュボードでの可視化(7.7.1 / WF5 と接続)

| ビュー | 表示内容 |
|---|---|
| 個別ルーブリック | 児童1人の3観点 × A/B/C 自動判定 + 編集可能(教師の手動上書き) |
| クラス全体 | 3観点の達成度ヒストグラム |
| 経時変化 | 観点ごとに月次推移グラフ(Tremor で実装) |
| 観察メモ | 教師が手書きで観察コメントを記録(`teacher_observations` 後日追加候補) |

#### 19.4.7 ルーブリックの法的位置づけ

- tetsu-board のルーブリックは **教師の評価判断を補助する参考資料**(指導要録への記載判断は教師に委ねる)
- 自動判定はあくまで「目安」、最終評価は教師が観察と総合し決定
- 利用規約・プライバシーポリシー(18.8) に「評価データの取り扱い」を明記

### 19.6 効果測定方法(C-16 確定)

#### 19.6.1 測定の3層モデル

| 層 | 手法 | 主担当 |
|---|---|---|
| ① 量的(自動) | tetsu-board 内の `quiz_attempts` 集計、ふくしゅうリスト消化、滞在時間 | 自動 |
| ② 形成的(授業中) | 教師観察、ルーブリック手動入力、振り返りシート | 教師 |

#### 19.6.2 事前事後テスト設計(Phase 9 パイロットで実施)

| 教科 | 事前テスト | 事後テスト | 採点 |
|---|---|---|---|
| 地理 | 12問(地名・地形・産業) | 同形式12問(別問) | 教師 |
| 経済 | 10問(値段・買い物・概念) | 同形式10問 | 教師 |
| 公民 | 8問(きまり・社会) | 同形式8問 | 教師 |
| 情報 | 8問(データ・機器) | 同形式8問 | 教師 |

| 数学 | 12問(数量・図形・確率) | 同形式12問 | 教師 |

#### 19.6.3 測定指標(Phase 9 パイロットで検証)

1. **教育的妥当性**: tetsu-board の合理的配慮設計と学習指導要領との整合性は妥当か?
2. **C層適合性**: 知的障害・発達系の児童生徒にとって本当に「学べる」「楽しい」設計か?
3. **倫理的安全性**: 児童データ取扱・同意フロー・難易度判定アルゴリズムに倫理的懸念はないか?

#### 19.7.5 倫理審査(IRB) との関係

| ケース | 対応 |
|---|---|
| 学校都合での中断 | データはその時点まで集計、報告書に明記、謝礼は実施分のみ按分 |
| 児童の参加同意撤回 | 該当児童のデータ即時削除(18.6 整合)、クラス全体は継続 |
| tetsu-board 側の致命的バグで運用不可 | 全校一時停止、修正後再開 or 期間延長協議 |
| パイロット数が 1校 まで減少 | Go/No-Go 判定で「データ不足」として Phase 9 延長(20.9) |

### 20.4 倫理審査(IRB)経路

#### 20.4.1 経路の選択肢

19.7.5 の方針を具体化:

| 経路 | 適用条件 | 利点 | 困難 |
|---|---|---|---|
| ① 大学共同研究 + 大学 IRB | 大学研究者と共同研究化(19.7.3 ⑤ への前倒し) | 制度的厳密性・公的承認 | 共同研究契約・IRB 申請に3〜6ヶ月 |
| ② 学校内倫理委員会 | 学校に独自倫理委があり、外部研究受入実績あり | 学校単位で完結 | 学校による |
| ③ 自治体教育委員会の承認 | 学校設置者の承認で代替 | 公的経路 | 自治体ごとに手続き異なる |
| ④ 自主アセスメント | 上記が困難な小規模 / 民間施設 | 即時着手可 | 学会発表時の倫理性証明が弱い |

第1波は ④ 自主アセスメント + 同意フロー厳格化で開始、第2波で ① 大学 IRB を経由する方針が現実的。

#### 20.4.2 自主アセスメント実施項目

| データ最小化 | 18.6 厳格化版項目のみ、目的外利用なし |
| 匿名化 | 報告書では学校名・個人名を完全匿名化、識別子はハッシュ |
| 児童の福祉優先 | ゲーム内で不快表現なし、難易度過剰時は撤退選択肢提示 |
| データ保管 | 暗号化保存、保管期間明示、研究終了後の破棄手順 |
| 利益相反 | 開発者と研究者の関係明示、謝礼の透明性 |
| 報告義務 | 重大事象(児童の精神的負担、データ漏洩懸念)の即時報告手順 |

### 20.8 リスクと対応策

#### 20.8.1 リスクマトリクス(発生確率 × 影響度)

#### 20.8.2 致命的リスク(R1, R6) の二重防御

- **R1 児童データ漏洩**: RLS + 暗号化 + 監査ログ + 漏洩時の72時間内通知体制(個人情報保護法 + 学校設置者 + 保護者)
- **R6 倫理逸脱**: 20.4 自主アセスメント + 大学 IRB + 内部チェックリスト(20.4.2)で各授業前確認

#### 20.8.3 撤退判定の基準

以下のいずれかで Phase 9 即時停止 + 再評価:

- R1 発生(児童データ漏洩確定)
- R6 発生(同意外利用 / 差別表現流出)
- パイロット校 3校以上で R2 が報告される(設計欠陥の可能性)
- 死亡・重傷事案(端末事故等、極めて稀)

#### 20.8.4 通常停止の基準

以下は Phase 9 として「成立せず」として、修正後 Phase 9'(第3波)で再実施:

- パイロット校 全数が R8 で期間 50% 未満消化
- 19.6.3 目標値の半数以上が未達(R10)
- 研究者書面レビューで「現状では妥当性に重大懸念」(R9 派生)
