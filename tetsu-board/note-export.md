# 鉄道テーマの教育ボードゲームを Claude と一緒に7フェーズで設計してみた

## はじめに

「鉄道テーマのボードゲームって作れるかな」――そんな何気ない問いから始まった対話が、最終的に **6ヶ月〜1年仕事の教育プロダクト構想** にまで発展した記録。

特支教育・夜間定時制で教員経験のある自分が、当事者性(ASD/ADHD)も活かしながら、Claude Code(対話型AI)と一緒に設計を詰めていったプロセスを残しておく。

## 完成した構想の一行コンセプト

> **授業のスキマで遊べる、5〜15分のクイズ駆動・視覚優先のボードゲーム。
> 知識定着がそのまま勝敗になる。仲間のプレイを見て学べる。**

鉄道ボードゲームの骨格を借りつつ、教育プロダクトとして再構成。第1弾ターゲットは **知的障害・発達系を含む特支/夜間定時制の生徒(C層)** とした。

## Phase 0:プロジェクト命名と方針合意

最初に「先行作品をリスペクトするけど独自路線」というスタンスを確認。空のフォルダに `tetsu-board` という仮称をつけてスタート。

ここで重要だったのは、 **「綿密な打ち合わせの上で設計図を作成する」** という姿勢を最初に決めたこと。手戻りを防ぐために、いきなり実装に入らず、設計フェーズを7つに分けて順番に確定させていく方針にした。

```
Phase 1: ゲームデザインの根幹
Phase 2: マップと移動の仕組み
Phase 3: 経済システム
Phase 4: イベント・カード・キャラ
Phase 5: プレイヤー構成
Phase 6: 技術スタック決定
Phase 7: 設計図化
```

## Phase 1:ゲームデザインの根幹

### 独自路線をどこで打ち出すか

5つの方向性候補を提示してもらい、4つを採用:

- ① AIキャラ路線(LLMで動的に喋る・煽る)
- ② 教育要素路線(地理・経済の学習)
- ④ 超短時間路線(10〜15分)
- ⑤ ローグライク路線(毎回違うマップ)

(③ 配信連動路線は今回は採用せず)

ここで重要な判断として **「教育に寄せる」** を選択。元教員の経験を直接活かせる路線で、市場の競合がほぼない領域だった。

### ターゲット層の選定

A層(小中)、B層(高校)、C層(特支/夜間定時制)、D層(大人)、E層(教員自身)の中から、 **C層を第1弾ターゲット** に選定。

選定理由は明快だった:
- 当事者経験(現場経験)が最も濃い
- 競合がほぼ存在しない
- C層向けに作った合理的配慮UIは、A/B層にも転用可能(下方互換性)

### 「教材性のレベル」を「濃い」に決める

教師用ガイド・授業設計案・評価ルーブリック・振り返りシートまでフル装備。これは個人開発のスケールを大きく超える決定だったが、市場差別化のために重要だった。

### 勝利条件と「9歳の壁」配慮

最も重要だった設計判断:

> **勝敗の通貨を「お金」から「正解」に変える**

従来作の勝敗は資産だが、これは抽象概念で「9歳の壁」(抽象概念の理解難)を直撃する。代わりに **クイズ正解数を主スコア** にして、コインや物件は副次的にした。

3つの合理的配慮原則を確立:
1. **抽象→具体**(コイン100枚は絵で表示、収益率5%は禁止)
2. **行間ゼロ**(比喩・敬語・慣用句を排除、「ご購入されますか?」→「かう?」)
3. **ビジュアライズ強化**(動きと絵で理解させる、文字より演出)

### プレイモードの2軸構成

| モード | 時間 | ターン数 |
|---|---|---|
| アイスブレイク | 5〜15分 | 12〜36ターン |
| 余暇エンドレス | 〜1時間 | 240〜360ターン(20〜30年) |

余暇モードは20〜30年プレイ前提なので **セーブ・再開機能を必須** に。

## Phase 2:マップと移動の仕組み

### カイロソフト風の架空駅 + 実在駅副表示

最初の議論では「実在駅 vs 架空駅 vs ハイブリッド」だったが、最終的に **「カイロソフト風の架空駅をメインに、実在駅は括弧で副表示」** という独自解に落ち着いた。

```
[ON: 漢字+ルビ + 実在駅併記]
苺の谷駅(愛媛・松山駅)

[OFF: 親しみ重視]
いちごのたに駅
```

副表示はON/OFF切替可能で、C層初期値はOFF、A/B層初期値はON、いつでもタップで参照可能。

### マップ形状:円環+少数分岐

従来作の複雑な日本地図を C層向けに簡略化。15〜20駅の円環ループに3箇所の2択分岐を配置。

### αβγハイブリッド分岐

驚いた判断のひとつ。3箇所の分岐に、4種類のリスクリターン構造をランダム割当する設計に決まった:

```
パターン1: 近道&危険 vs 遠回り&安全(王道)
パターン2: 近道&安全 vs 遠回り&危険(直感を裏切る)
パターン3: 両方危険(緊張感、思い出に残る瞬間)
パターン4: 両方安全(休憩ターン、認知的に休める)
```

特に **パターン4(両方安全=休憩ターン)** は意外な発見だった。「ずっと真剣判断」が辛い児童・生徒のために、 **意図的な休憩がゲーム内に存在する** という設計。

### 漢字+ルビ統一(全UI)

「平仮名は全て漢字に変換(漢字+ルビ)」 と決定。理由:
- 漢字+ルビ = 読めない子も読めるが、漢字を目に入れることで学習効果あり
- 切替で純ひらがな = 漢字の認知負荷で詰まる児童・生徒は除外可能

これは後で全UI設計の核となる原則になった。

## Phase 3:経済・スコア詳細

### 駅カテゴリ=産業構造

各駅カテゴリと物件種を1対1対応に:

| 駅カテゴリ | 物件種 | 産業 |
|---|---|---|
| 田畑・里山系 | 畑物件 | 第一次 |
| 海・港系 | 漁港物件 | 第一次 |
| 工業系 | 工場物件 | 第二次 |
| 都市系 | お店物件 | 第三次 |
| 観光系 | ホテル物件 | 第三次 |

これにより **社会科の産業構造学習が自然にゲームに組み込まれる** 構造になった。

### 3部門制スコアの発見

最大の論点は「クイズ正解数 vs 資産」のスコア重み付け。3案あったが、最終的に **「完全分離・3部門制」** を選択:

- 🏆 クイズ王(正解数1位)
- 💰 金持ち王(資産1位)
- ⭐ 総合王(両方の合計1位)

これによる効果:
- 計算が苦手な児童も「正解で1位」になれる
- 物件運営が得意な児童も「金持ちで1位」になれる
- 自己効力感を最大化(特支教育の核)
- A/B層展開時も同じ仕組みで対応可能

「点数で序列化する」のではなく、 **「複数の評価軸で誰もが何かで輝ける」** という設計思想。

### %表現の扱い

ここで興味深い議論があった。当初は「%禁止(9歳の壁)」だったが、議論の末 **「整数表示+%併記オプション(ON/OFFトグル)」** に。

これは漢字+ルビと同じパターンで、 **「合理的配慮としてのオプション」** に統一できた瞬間だった。

## Phase 4:イベント・カード・キャラ

### 災難キャラ:「悪意なく困った性格」+「少し怖い」

鉄道ボードゲーム先行作品の災難キャラ(独自)的な存在を、3種類のキャラクターで設計:

```
コインモグラ: 1ターンに1コイン食べる
ねぼすけ犬:  1ターン休み
おっちょこちょい星人: サイコロの目が半分
```

特徴:
- 進化なし(C層配慮で複雑化を避ける)
- 「悪意なく困った性格」(怖くない)
- でも「少し怖さ」も加える(可愛さ80% + 不気味要素20%)
- 離れる条件は **「クイズ3連続正解で離脱」**(教育的成功体験)

### 教育カード3種(独自要素)

先行作品のカードに加えて、教育カードを追加:

- 💡 ヒントカード:答えのヒントを表示
- 🤝 おてつだいカード:AIキャラと一緒に答える
- 🔁 もう一度挑戦カード:不正解時に再挑戦

「もう一度よむカード」(問題文を再読み上げ)は、 **「リピートしても理解できないものはできない」** という当事者の知見から削除。代わりに「もう一度挑戦」(失敗からの回復体験)に差し替え。

### 攻撃系カード(牛歩等)は削除

**「嫌がらせ要素は自己効力感を損ね、教育場面で適切でない」** との判断。

### CPU対戦キャラ:季節シリーズ4キャラ

```
サクラくん 🌸春 - 数学得意
ヒマワリくん ☀️夏 - 理科得意
モミジちゃん 🍁秋 - 国語/社会得意
ユキちゃん ❄️冬 - 家庭/情報得意
```

「興味のある教科のキャラを選ぶ動機 → 学習動機に直結」という工夫。

### ビジュアル:カイロソフト風ピクセルアート

絵柄テイストは **「ピクセルアート系(カイロソフト風)」** に決定。ゆるキャラ系・絵本系も候補だったが、駅名のカイロソフト風ネーミングと整合する選択。

### サウンド:全要素独立切替

BGM / 効果音 / キャラ音声 / 文字読み上げを **すべて独立にON/OFF切替可能**。初期値は控えめ。ASD/ADHD当事者は感覚過敏に個人差があるため、個別調整できる柔軟性を最優先した。

## Phase 5:クイズシステム詳細

### 設計原則「イニシアチブはユーザ」の発見

このフェーズで最も重要な発見。動的難易度の制御権限を議論する中で、 **「イニシアチブはユーザ(幼児・児童・生徒)が持つ」** という原則が明文化された。

含意:
- 自動調整するが、本人が「もっと頑張りたい/ゆっくりやりたい」と選べる
- 教師は監督・必要時介入のみ
- 自己決定権を尊重(特に当事者主体・自己効力感の核)
- ASD/ADHD当事者支援としても整合

これは後にPhase 7まで貫通する設計原則となった。

### 消去法UIの採用

クイズの回答形式は **3択選択(絵+文字)** を選択(2択は易しすぎ、4択はハードル高い)。

そして「消去法で選択する」という当事者の言葉から、 **消去法UI** を実装機能として採用:

```
短タップ: 「ちがう」(×を付ける、暗くなる)
長タップ or「これだ」ボタン: 回答確定
「もどす」: ×を取り消し可
```

「これは違う、なぜなら~」と論理的に考える練習ができる。

### 教育ダッシュボードを必須機能に

「不正解時の処理」を学習接続版(記録→後日類題出題)にすることが決まった瞬間、 **教育ダッシュボードが必須機能(MUST)** に格上げされた。

これにより、技術スタック選定の重みが大きく変わる:
- データベース必須
- 認証システム必須
- データ可視化UI必須
- 児童・生徒側にも「ふくしゅうリスト」が見える(イニシアチブ原則)

### 制限時間もユーザ選択

クイズの制限時間も「ゆっくり/ふつう」をユーザが選択。教師が枠(上限・下限)を設定できる構造。

すべてのアクセシビリティ設定が **「ユーザが選ぶ・教師が枠を設定する」** という統一パターンに揃ってきた。

### 問題作成方針

「現在は個人開発」というスケール前提で、段階展開:
- 第1弾: AI生成 + 自作レビュー(数百問)
- 第2弾: 教師コミュニティ参加(千問規模)
- 第3弾: ユーザー投稿(数千問規模)

## Phase 6:技術スタック決定

### Next.js + Vercel + PWA

最初は「Next.js + Capacitor wrap でiOS/Androidネイティブアプリ化」を検討したが、 **「ネイティブでなくPWAだ」** という方向修正。

理由:
- 個人開発の負荷軽減
- App Store / Play Store配布不要(教育機関導入の障壁低減)
- 「余白を設ける」方針(後からCapacitorでネイティブ化可能)

3形態(教室1台 / 教師PC+生徒タブレット / オンライン)はすべてWebアプリで対応。

### Supabase オールインワン

DB(Postgres)・Auth・Storage・Realtime・Edge Functions を一括採用。

決め手:
- Vercel Marketplaceで1クリック導入
- Postgres でSQL集計クエリ◎(教育ダッシュボード向き)
- AWS東京リージョン選択可(児童データの国内保管)
- Row Level Securityで「自分のデータのみ・担当クラスのみ」を宣言的保証

### PixiJS + ビジュアルサイズ調整(アクセシビリティ機能)

ピクセルアート描画は PixiJS + @pixi/react を採用。ここでも **「ビジュアルサイズもアクセシビリティ設定で調整可能に」** という独自の判断が入った。

統一のアクセシビリティ設定:
- 文字(漢字+ルビ ON/OFF、ふりがな大きさ)
- ビジュアル(ゲーム画面・UIボタンのサイズ)
- 音(BGM/効果音/キャラ音声/文字読み上げ独立切替)
- ペース(制限時間、動的難易度)
- 数字表現(%ON/OFF)

### Vercel AI Gateway + Claude

LLM選定は「Claude主軸 + AI Gateway経由で複数切替可」で決定。ロックインなし、後からGemini/GPTへ切替可能(余白を設ける方針)。

### 教育ダッシュボード:Shadcn/UI + Tremor

汎用UIはShadcn/UI、ダッシュボード特化はTremor。両者のハイブリッドで個人開発負荷を軽減。

## Phase 7.5:法務リスク回避と法的論点の整理

35項目のうち最も致命的だったのが **法務・コンプライアンス系7項目**。先行作品(鉄道テーマのボードゲーム)との類似性をリーガル観点で整理し、独自作品としての位置づけを確立する作業を行った。

### 主要な法的論点

#### 1. 商標法

| 商標(推定) | 推定権利者 | 当該プロジェクトとの関係 |
|---|---|---|
| 「桃太郎電鉄」「桃鉄」 | コナミデジタルエンタテインメント株式会社 | 完全独自名称(`tetsu-board`)のため抵触なし |
| 「ボンビー」「キングボンビー」「ハリーボンビー」 | コナミ(キャラクター標章と推定) | 完全独自キャラ名のため抵触なし |

→ **商標侵害リスクなし**。ただしマーケ表記で「桃鉄風」等を使うと **商標的使用** と見なされるリスクがあるため削除。

#### 2. 著作権法(アイデア・表現二分論)

| 著作権で保護される(表現) | 保護されない(アイデア) |
|---|---|
| キャラクター絵柄・固有名称 | ゲームシステム・ルール |
| ストーリー・プログラムコード | 抽象的なゲームメカニクス |
| マップの具体的レイアウト | アルゴリズム |

→ サイコロ・物件購入・独占ボーナス等の **メカニクス類似は著作権の保護対象外**。完全独自のキャラ絵・固有名称なので **著作権侵害リスクなし**。

#### 3. 不正競争防止法

| 条項 | 内容 | 本プロジェクトとの関連 |
|---|---|---|
| 第2条1項1号(周知表示混同惹起) | 周知の他人の商品等表示と類似で混同を生じさせる行為 | 「桃鉄風」マーケ表記が混同惹起の可能性 → 削除済 |
| 第2条1項2号(著名表示冒用) | 著名な他人の商品等表示の冒用 | 同上 |
| 第2条1項3号(商品形態模倣) | 他人の商品形態を模倣 | 該当なし(独自表現) |

→ **不正競争防止法上のリスクなし**(対応済)。

#### 参考判例

| 判例 | 裁判所/年 | 教訓 |
|---|---|---|
| パックマン事件 | 東京地判 昭和57年(1982) | 画像・プログラムは保護、メカニクスは保護対象外 |
| たぬきの大冒険事件 | 東京地判 昭和62年(1987) | 表現の本質的特徴の同一性で判断 |
| **釣りゲータウン2事件** | **知財高判 平成24年(2012)** | **メカニクス共通+画面構成類似でも、具体的表現が独自なら著作権侵害成立は困難** |

特に釣りゲータウン2事件(知財高判 H24.8.8)が本件には参考になった。グリーが DeNA を著作権侵害で提訴した事案で、二審は侵害否定。「アイデアではなく具体的表現の比較」が判断ポイント、という判旨。

## Phase 7.5 確定: ビジネス・法令整合の8論点

法務リスク回避の整理に続き、Phase 7.5 では **第三次審査で発覚した 35項目のうち A系(法務残り)・B系(ビジネス)の論点** をまとめて整理した(v1.8 で確定)。

### 課金モデル: フリーミアム + B2B(3層構造)

C層(特別支援学校・夜間定時制)を主軸に据えると、家庭の支払能力にばらつきがあるため、 **「無料で触れる」入口が決定的に重要** だと判断した。

| 層 | 構成 |
|---|---|
| **Free(個人)** | 1教科(地理) / 10駅 / ソロ+ホットシート2人 / 合理的配慮全機能利用可 / ローカル保存のみ |
| **プレミアム(個人)** | 全教科解放 / 全駅マップ / オンライン同期マルチ / 親モード / Supabase 同期 |
| **教育機関ライセンス** | フル機能 + クラス管理 + 教師ダッシュボード + 学年集計、生徒数比例の年額契約 |

特筆すべき点: **合理的配慮機能は Free 層でも全機能利用可** とした。教育的公平性の観点で、配慮機能を有料化することは設計思想に反する。

LLM ランタイムコストが低い設計(問題生成・キャラセリフは事前バッチ)のため、無料層を厚くしても破綻しない。具体価格はパイロット後に実測ベースで決定。

### ユーザーサポート体制: 個人と教育機関を分離

| 対象 | 構成 |
|---|---|
| **個人** | アプリ内お問い合わせ → GitHub Issues 連動、メール、アプリ内ヘルプ(動画+ルビ)、ステータスページ |
| **教育機関** | 専用窓口、Slack/Teams 連携可、教師ダッシュボードに「クラス全体の不具合報告」ボタン |

C層配慮として、 **「うまくいかなかった」ボタンを大きく、ルビ付きで設置**。SLA は実測後決定。

### 競合分析: 独自ポジション5軸

国内: 桃太郎電鉄教育版・すらら・スマイルゼミ・Qubena・Kahoot!
海外: Prodigy Math・Duolingo・ClassDojo・BrainPOP・DragonBox

比較の結果、 **tetsu-board の独自ポジションは以下の5軸** で確定:

1. **「特支・C層特化」**: 国内外で本格対応する競合なし(すららは部分対応)
2. **合理的配慮を設計の根幹**: 障害者差別解消法準拠を看板に出来る数少ない教育ゲーム
3. **教科横断 × ボードゲーム性**: Prodigy Math は教科単独、Kahoot! は形式単独。tetsu-board は両立
4. **イニシアチブはユーザ原則**: 教師による枠設定 + 児童主導という独自設計
5. **法務クリア独自IP**: 桃鉄教育版に対して家庭・自治体導入両面で展開可能

特に最大の参考事例は **Prodigy Math**(米/加)で、 **ゲーム性 + アダプティブ + フリーミアム + 学校導入の組合せが実際に成立している** という前例があるのは励みになる。

### 配布チャネル: B2C 強化 + 放課後等デイサービス(B2B2C 中間)

ここで重要な発見があった。当初は「学校 vs 個人」の二分法で考えていたが、 **放課後等デイサービスがその中間にある最重要チャネル** だと気づいた。

放デイは:
- 家庭の支払いだが福祉契約
- 特支系児童が集まる
- 月額契約モデルとの相性◎
- 施設運営者の決定で導入可能(学校のような複雑な意思決定プロセスなし)

| Stage | 内容 | 主チャネル |
|---|---|---|
| Stage 0: プレリリース | 開発記録発信、ベータテスター募集 | note・X・YouTube |
| Stage 1: MVP公開 | Web/PWA同時公開、家庭での試用促進 | SEO・YouTube・X・保護者コミュニティ |
| **Stage 2: 放課後等デイ展開** | 月額契約モデルで施設に展開 | 福祉系媒体・直接営業 |
| Stage 3: 学校導入 | 家庭/放デイ実績を学校提案に活用 | 教委ルート・学会発表 |
| Stage 4: 自治体一括 | 全市町村展開 | プロポーザル・首長提案 |

B2C 強化チャネル: SEO・YouTube・X・LITALICO発達ナビ等の保護者コミュニティ・Instagram/TikTok・ウェビナー。Capacitor でのアプリストア配信は将来の余白。

### 児童データ同意フロー: 3シナリオ + 厳格化

| シナリオ | 同意取得者 |
|---|---|
| 個人家庭 | 保護者(法定代理人として代理同意) |
| 学校導入 | 学校が契約範囲で保護者から取得 |
| 放課後等デイ | 施設が保育・福祉契約の範囲で保護者から取得 |

個人情報項目は **さらに厳格化** した:

- **ニックネームも任意**(未入力時「プレイヤー1」等で運用)
- **学年も任意**
- 保護者メールはオプトイン
- 顔写真・音声・位置情報・生年月日は設計から完全除外
- Free 層はローカル保存のみ(Supabase 同期なし)

「個人特定不能でも、ゲームとして成立する」設計を貫く。

### COPPA / GDPR-K: 設計余白として識別子を残す

第1弾は日本国内法のみ準拠。COPPA(米国 13歳未満)・GDPR-K(EU 16歳未満)は将来の国際展開時のみ対応。

ただし、 **設計に「年齢区分・地域別同意フロー」の識別子を余白として残す**:
- `region_code`(JP/US/EU/...)
- `consent_scope`(parental / institutional)
- `consent_version`

将来 COPPA・GDPR-K 対応を入れる際にスキーマ変更不要な余白設計。

### 規約・プライバシーポリシー: 3系統で骨子化

個人・教育機関・放課後等デイの3系統で骨子を作成。 **個人と教育機関でデータ取扱責任の所在が違う** ことが要点:

- 個人: 運営者が取扱責任
- 教育機関: **学校が保護者同意取得を行う**、運営者は処理委託
- 放課後等デイ: 施設が同意取得、児童福祉法準拠を明示

### 学校教育法・障害者差別解消法の整合

| 法令 | 整合性 | 対応 |
|---|---|---|
| 学校教育法 | ◎ 補助教材位置づけ(検定教科書の代替ではない) | ✅ |
| 一般学習指導要領 | ○ 教科横断補助教材 | ✅(詳細マッピングは Phase 8) |
| **特別支援学校学習指導要領** | **◎ 全面実装** | ✅ |
| **障害者差別解消法**(2024年民間事業者義務化) | **◎ 合理的配慮が根幹** | ✅ |

特に **障害者差別解消法第8条2項(2024年4月から民間事業者にも合理的配慮義務化)** に対して、tetsu-board は **設計の根幹に合理的配慮を実装** しているため、法施行と完全に整合している。

## 設計を貫く4つの原則

最終的に、全フェーズに貫通する4つの原則が確立した:

### 1. 合理的配慮(障害者差別解消法準拠)

漢字+ルビ・サウンド独立切替・ビジュアルサイズ調整・制限時間ユーザ選択・動的難易度・%表現切替・抽象→具体・行間ゼロ・ビジュアライズ強化を **全UI/UX に実装**。

「C層配慮」と呼ばずに「合理的配慮」と呼ぶことで、 **特定層への配慮ではなく、誰もが受けられる権利** として位置づけた。

### 2. 9歳の壁対策

合理的配慮の中核となる対策。抽象概念の理解が困難な発達段階の児童・生徒のために、抽象→具体・行間ゼロ・ビジュアライズ強化を徹底する。

### 3. イニシアチブはユーザ

すべてのインタラクションで児童・生徒(=ユーザ)が主導権を持つ。教師は監督・必要時介入・枠の設定者。動的難易度・制限時間・ヒント要求・もう一度挑戦・ふくしゅうリスト――すべてユーザが選べる。

### 4. 余白を設ける

段階的拡張可能なアーキテクチャ・スケジュール。MVPに必要な最小限+後で必要になりそうなものは「余白」として残す。「完璧主義にならず、コアを作り、後から育てる」設計思想。

## 学び・気づき

### 1. AIとの協働は「対話の質」で決まる

最初に「綿密な打ち合わせの上で設計図を作成する」と決めたことが、すべてのフェーズの質に直結した。各フェーズを順番に確定させていく姿勢が、後の手戻りを防いだ。

### 2. 「一任シグナル」と「1問ずつ」のバランス

「一任します」と言える局面と、「1問ずつじっくり詰めたい」局面がある。これは ASD/ADHD 特性的に重要で、バッチ質問だと処理過多で取りこぼしが起きる。AI側がこれを学習して使い分けてくれたことで、対話の質が上がった。

### 3. 当事者経験が設計に直接活きる

特に印象的だったのは:
- 「ハードルは高すぎず低すぎず、消去法で選択します」 → 消去法UIの採用
- 「リピートしても理解できないものはできない」 → もう一度よむカード削除
- 「イニシアチブはユーザ」 → 全UIに貫通する原則
- 「合理的配慮にした方が良いかな」 → 法令準拠の用語統一

これらは元教員 + 当事者でなければ出てこない判断だった。

### 4. 「余白を設ける」設計思想

完璧を最初から目指さず、段階的に育てる設計。コアエンジン+UIスキン+コンテンツパック構造、Phase 1〜3展開、教育ダッシュボード段階展開、問題作成段階展開――どれも「余白」を意識した設計になった。

### 5. 第三次審査の重要性

「全て大丈夫ですか?遺漏ありませんか?」と何度も問うことで、35項目の遺漏が発覚した。 **「人間の目はOK。AIから見て遺漏は」** という問いかけは、AI協働における重要な技法かもしれない。
