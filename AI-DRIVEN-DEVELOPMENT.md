## 1. 背景・課題意識

### 1-1. 暗黙知の「教室への閉じ込め」
教育現場で16年蓄積した知見 ——「生徒がどこでつまずくか」「どの順序で教えれば定着するか」「失敗をどう扱えば学習者が再挑戦するか」—— は、本来きわめて再利用価値が高い。しかしこの知見は属人的・暗黙的であり、教室という時空間に閉じてきた。これをソフトウェアとして外在化できれば、教員一人の手が届く生徒数を超えて知見が作用しうる。

### 1-2. これまでの壁＝実装力
暗黙知をソフトウェア化する最大のボトルネックは、ドメイン知ではなく**フルスタック実装力**だった。要件定義・DB設計・認証・RLS・本番デプロイ・運用までを一人で回すには、従来は専業エンジニアの分業が前提だった。

### 1-3. 掛け合わせの新規性
本取り組みの新規性は **「教育ドメイン知 × フルスタック実装 × AIエージェント（AI駆動開発）」** の三者の掛け合わせにある。
- 従来の教材デジタル化は「教育者が仕様を書き、エンジニアが作る」分業型。
- 本取り組みは、**教育者本人がAIエージェント（Claude Code 等）を実装パートナーとして、要件定義から本番運用までを一人で貫通する**。ドメイン知の保有者と実装意思決定者が同一人物であるため、「なぜこの機能か」が検定シラバス・教育メソッドから直接逆算され、仕様の翻訳ロスが起きにくい。

この「ドメイン専門家がAIで実装力の壁を越える」構図は、教育に限らず非・専業エンジニアの専門家全般に展開可能な方法論として整理しうる。

## 2. 研究上の問い／主張

### 2-1. リサーチクエスチョン
- **RQ1**: 非・専業エンジニアの教育者が、AI駆動開発によって「要件定義 → 実装 → 検証 → 本番運用」を一人で完遂できるか。
- **RQ2**: その過程から、**再現可能な方法論**として何を抽出できるか（誰が真似ても同じ品質に近づける「型」は存在するか）。
- **RQ3**: 教育者ならではの非機能要件（アクセシビリティ・特別支援・失敗を罰しない設計）を、AI駆動でどこまで実装に落とし込めるか。また、それは差別化の核になるか。

### 2-2. 主張（クレーム）
1. **完遂可能性**: 個人・実証段階という限定つきだが、複数の教育系Webアプリを設計から本番デプロイ（Vercel）まで一人で到達させた（README記載のとおりipas-master / logic-riichiが本番稼働、計10本超のプロダクトが存在）。
2. **再現性の担保物**:「Git履歴 ＋ To-be要件定義（`docs/requirements.md`）＋ 設計書（Blueprint）＋ 自動テスト」の4点セットが、属人化しがちな個人開発を**追跡・再現可能な状態**に保つ。
3. **ドメイン知の優位**: AIは実装を高速化するが、「何を作るべきか（教育的妥当性）」は依然として人間のドメイン知が決める。AI駆動開発はドメイン専門家の価値を**置換せず増幅する**。

## 4. AI駆動開発のワークフロー（方法論の核）

本章が発表の中心。再現可能な「型」として箇条書きで提示する（ポスター・スライドで図解しやすい粒度）。

### 4-1. 「型」の全体像（5ステップ・ループ）
```
[1] 要件をTo-beで明文化     docs/requirements.md（P0/P1/P2・✅/🔄/🆕）
        │  ← 教育的妥当性・差別化軸を北極星として固定
        ▼
[2] ロジックを純関数に分離   lib/game/engine.ts 等（副作用ゼロ）
        │  ← 永続化/認証/Realtimeは後付けにできる構造で手戻り防止
        ▼
[3] AIに実装させる          Claude Code で生成→人間が受け入れ判断
        │
        ▼
[4] 自動テストで固定        vitest 回帰テスト（describe/it）
        │  ← 「直近実装の数え方を固定する」回帰の網
        ▼
[5] コミットで全工程を追跡    1機能=1コミット、履歴が設計判断の証跡
        │
        └────────── レビュー（AI多観点＋人間最終）→ [1]へ
```

### 4-2. 各ステップの具体（事実に基づく）
- **[1] 要件のTo-be明文化**: 各リポジトリに `docs/requirements.md` を整備。現状ではなく**あるべき姿（To-be）**を、優先度 **P0/P1/P2** と状態 **✅実装済 / 🔄強化 / 🆕新規** のタグで管理（実例： `logic-riichi/docs/requirements.md`、`otetsudai-bank/docs/requirements.md`、`tetsu-board/docs/requirements.md`、いずれも最終更新2026-06-19）。冒頭に「ポジショニング」「差別化軸（北極星）」を置き、全機能要件がそこに従う構造。
- **[2] 純関数へのロジック分離**: 手戻り防止の中核パターン。実例 `tetsu-board/lib/game/engine.ts` 冒頭コメント：
  > 「ゲームロジック（純関数）。… クライアントでも将来のサーバ実装でも共用できる。乱数だけ注入可能にして、サーバ移行時は `crypto.randomInt(1,7)` へ差し替える。」
  - 乱数源を引数注入（`rollDice(rng = Math.random)`）にすることで**テスト容易性とサーバ移行容易性を同時に獲得**。
  - 要件書にも不変原則として明記：「ゲームロジックは副作用なしの純関数（`lib/game/engine.ts`）に分離。永続化・認証・Realtimeは後付けでstoreのアクションをServer Actionに差し替えるだけにする」（`tetsu-board/docs/requirements.md` §0）。
- **[3] AI実装**: Claude Code / Cursorでコード生成。人間は受け入れ判断に集中。
- **[4] 自動テスト（vitest 回帰）**: 実例 `tetsu-board/lib/game/engine.test.ts` 冒頭：
  > 「engine.ts（移動・分岐・回答の純ロジック）の回帰テスト。直近実装した分岐の『歩数』と『通過コイン』の数え方を固定する。マップは generateMap に依存せず、手組みの最小 fixture で意図を明示する。」
  - `import { describe, expect, it } from "vitest"` で、`walk` / `continueFromBranch` / `judgeAnswer` / `calcScore` / `shortestDistance` / `shortestPath` / `destinationCandidates` / `pickDestination` を対象に回帰を固定。
  - **純関数化（[2]）があるからテストが書ける**という依存関係が重要。
- **[5] コミット履歴での追跡**: 1機能＝1コミットを基本に、コミットメッセージへ設計意図（例： 親リポの `feat(tetsu-board): 分岐をID参照化(手戻り防止)`）を残す。履歴自体が「いつ・なぜその設計判断をしたか」のドキュメントになる。

### 4-3. AIによる多観点レビュー
人間の最終受け入れの前段に、AIに「セキュリティ（RLS・キーの扱い）」「アクセシビリティ」「型安全」など観点を変えてレビューさせる。例：「サーバAPIでanon_keyを使わずservice_role_keyを使う」といった是正が運用知見として蓄積されている（おこづかいクエストv0.4のRLS強化・PIN暗号化はこの種のレビューの産物）。

## 5. 事例（プロダクト横断・AI駆動開発の観点で各1段落）

各事例を「**何をAIに任せ、何が難しかったか**」で記述する。

- **IPAS-Master（ITパスポート対策・v9／本番稼働）**: 500問ドリル・14分野フィルタ・損益分岐点シミュレーター・2進数変換ツール・NextAuth(GitHub OAuth)・Supabase学習履歴・レーダーチャート。AIには定型的なCRUD・グラフ描画・認証配線を任せやすかった。難しかったのは**問題分野の分類体系（14分野）と「たとえ話付き用語解説」の教育的設計**で、ここは検定シラバスと教員の暗黙知に依存し、AIに丸投げできない部分。

- **Logic-Riichi（麻雀×アルゴリズム学習・v30／本番稼働）**: 待ち牌クイズ＋アルゴリズム実装問題（選択式→虫食い→自由記述の段階的足場外し）。**待ち牌判定ロジックは純関数化と相性がよく**、AIによる実装・最適化（ビット演算）の恩恵が大きい領域。難所は「段階的足場外し」を学習設計として破綻なく成立させるUI/問題系列の設計と、全漢字ルビ等のアクセシビリティ。To-be要件では動的OGPと公開APIを次期P0に据えている。

- **おこづかいクエスト Web（otetsudai-bank・v0.4／本番β）**: お手伝い→報酬→つかう/ためるの金融教育。AIは家族/ユーザー/タスク/ウォレット等9テーブルのDB操作・サービス層分離（`lib/services/`）を高速に生成。難所は**セキュリティ非機能要件**——全9テーブルRLS、`get_my_family_id()` による家族単位アクセス制御、pgcrypto+bcryptのPIN暗号化（平文非保持）。AI生成物の**正しさ検証（特にRLSの抜け）**が最も神経を使った。

- **おこづかいクエスト Mobile（otetsudai-quest-mobile・React Native/Expo・Phase 3）**: Web版要件を共通コアとして共有し、プラットフォーム固有要件を別 `docs/requirements.md` に分離。難所は**Web版とモバイル版の「常にセット更新」運用**（改修の二重実装漏れ防止）と、ネイティブでの全漢字ルビ表示（marginによる余白調整など、iOS固有の細かな知見が必要）。

- **tetsu-board（鉄道テーマ教育ボードゲーム・2027リリース目標）**: 本書の方法論を最も明確に体現。**純関数エンジン（`lib/game/engine.ts`）＋ vitest回帰テスト**で、シード式マップ生成・移動・分岐・クイズ判定・決算を副作用なしで実装し、後から永続化/Realtimeを差し込める構造にしている。難所は教育コンテンツ量・ゲームバランス、そして**法務/事業リスク**（桃鉄教育版が全国小学校の約35%に無償導入済みという競合状況、お邪魔キャラの独自化、地図データの出典表示義務）。技術より「何を作るか・作ってよいか」の判断が支配的な事例。

- **Report-Hub（日報統合・v1／本番）**: 日報・工数削減トラッカー・暗黙知ナレッジベース（5テーブル、RLSで自分のデータのみ）。定型業務アプリでAIの恩恵が素直に出る一方、「暗黙知を蓄積する」というコンセプト自体は教育者の関心（知見の外在化）の反映。実務系（asset-management / roi-simulator / team-signage等）は拡散性は低いが、AI駆動で短期に量産できる実証群として方法論の再現性を裏づける。

## 7. 教育・アクセシビリティの独自性

教育者ならではの**非機能要件**を、AI駆動でどう実装に落としたか。

- **全漢字ルビ**: 未就学〜小学生・特別支援を想定し、全漢字にふりがな（共通部品化。tetsu-boardの `RubyText`、Logic-Riichiの全漢字ルビ）。実装知見として「ルビと漢字の距離はiOSで `marginTop:-2`」「ルビには `paddingVertical` 禁止、`margin` で余白」「同音異義語はDB側で漢字に直す」などの細かな失敗が蓄積した。AIに任せきれず人間の検証が必須だった領域である。
- **UDフォント**: BIZ UDPGothicを優先（ユニバーサルデザイン書体）。
- **特別支援対応**: おこづかいクエスト・tetsu-boardともに「特別支援・発達障害ファースト」を**差別化の北極星**に据える（手順分解・選択肢数・制限時間の可変、読み上げ）。tetsu-boardは「特別支援・夜間定時制・教科横断」を、無償の桃鉄教育版に対する差別化軸として要件に明記。
- **失敗を罰しない設計**: 誤答を罰点でなく「ふくしゅう（再出題）」として扱い、再挑戦を促す。SDL（自己主導型学習）・再認法→再生法（選択肢→虫食い→自由記述の足場外し）を全アプリ共通方針に。
- **色覚多様性・モーション配慮**: 色＋アイコン併用（✓/✗・▲/▼）、`prefers-reduced-motion` 対応、最小タッチターゲット44px、スキップリンク・フォーカスインジケータ。

要点： これらは**教育者だから要件として挙げられた**もので、AIは実装を担うが「何をアクセシブルにすべきか」の判断はドメイン知が起点。新規性の主張（第1章）を最も支える層。

## 8. 考察・限界・倫理

- **AI生成物の正しさ検証**: AIは速いが正しいとは限らない。とくに**RLS・キーの扱い・PIN暗号化など「間違うと安全性が壊れる」層**は人間検証が必須（おこづかいクエストv0.4はこの検証を経た強化）。純関数化＋回帰テストは、この検証コストを下げる工学的手段。
- **著作権・知財**: AI生成コード／アートの権利関係、および**既存作品との距離化**（tetsu-boardのお邪魔キャラ独自化、地図データの出典表示義務、商標J-PlatPat調査）は、技術より重い論点。教育市場では「学校の授業利用を守る著作権法35条は自社プロダクトの開発・配布を正当化しない」点に注意。
- **属人性 ↔ 再現性のバランス**: ドメイン知が属人的であるほど成果は独自になるが、再現性は下がる。本取り組みは「要件定義・設計書・テスト・Git履歴」で再現性を補うが、**完全な再現は原理的に困難**（同じAIでも出力が揺れ、ドメイン判断は人に依存）。
- **実証段階であることの限界**: ユーザー数・学習効果のエビデンスはまだない。本書の主張は「一人で完遂できた／方法論を抽出できた」までで、「教育効果がある」はこれからの検証課題。
- **倫理**: 子ども向け（おこづかいクエスト）の安全設計として、カード非発行・換金不可・対外決済なしを構造的に徹底し、規制（資金移動業・前払式支払手段）を回避。AI伴走（AIチャット）を子どもに使わせる際の安全配慮も継続課題。

## 9. 発表ストーリー案

### 9-1. 発表の流れ
1. **背景**（5%）: 16年の暗黙知の「教室への閉じ込め」と実装力の壁。
2. **問い**（5%）: 非エンジニア教育者がAI駆動で完遂できるか／再現可能な方法論は何か。
3. **手法**（35%）: 第4章の5ステップ・ループ（要件To-be → 純関数分離 → AI実装 → vitest回帰 → コミット追跡）。`engine.ts`/`engine.test.ts` を実コードで提示。
4. **結果**（25%）: プロダクト群（本番稼働多数）と、Git履歴・要件・テストという再現性の担保物。定量表（6章、集計後）。
5. **独自性**（15%）: 全漢字ルビ・UD・特別支援ファースト・失敗を罰しない設計（第7章）。
6. **考察**（10%）: 限界（実証段階）・倫理（知財/子ども向け安全）。
7. **今後**（5%）: 教育効果の検証、製品化（おこづかいクエスト・Logic-Riichiを速やかに、tetsu-boardは2027）。
